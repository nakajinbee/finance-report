# リクルートホールディングス 5年分データ取得メモ

## 結論：使うAPIは2つ

EDINET API（Version 2）には以下の2つしかない。書類を「検索」するAPIは無いので、
日付ベースで一覧を取得 → 目的の書類を絞り込み → 書類取得、という2段階の手順を踏む。

| API | エンドポイント | 役割 |
|---|---|---|
| ① 書類一覧API | `GET /api/v2/documents.json` | 指定した**1日分**の提出書類一覧を取得する |
| ② 書類取得API | `GET /api/v2/documents/{書類管理番号}` | ①で見つけた書類（ZIP/PDF/CSV等）の実データを取得する |

ベースURL：`https://api.edinet-fsa.go.jp`

**注意：企業名や証券コードで直接検索するAPIは存在しない。**
そのため「①を日付を変えながら何度も叩いて、レスポンスの中からリクルートの行を自分でフィルタする」という使い方になる。

---

## 事前準備

1. EDINET（https://disclosure2.edinet-fsa.go.jp/ の関覧サイト）でアカウント作成
2. APIキーを発行（`2-3-2-1 APIキーの発行`）→ `.env` に `EDINET_API_KEY` として保存
   - 設計書 `docs/design/cycle1_design.md` の「APIキー管理：サーバーサイド環境変数（`.env`）」と一致

## Step 1：リクルートのEDINETコードを特定する ✅実機検証済み

書類一覧APIのレスポンスに `edinetCode` と `secCode`（証券コード）が両方含まれるので、
**証券コード `6098` で最初に1回だけ絞り込んで edinetCode を確定**させれば、以降はそれを使い回せる。

```
GET https://api.edinet-fsa.go.jp/api/v2/documents.json
    ?date=2024-06-20
    &type=2
    &Subscription-Key={APIキー}
```

- `date`：適当な直近の営業日（有価証券報告書が集中する6月中旬あたりが狙い目）

### 実機検証結果（2026-07-19実施）

`backend/.env` の `EDINET_API_KEY` を使って `date=2026-06-19` で実行 → HTTP 200、`resultset.count: 1116`件、正常応答を確認。

レスポンス内の `secCode` はメモ作成時の想定（4桁 `"6098"`）と異なり、**実際は5桁で末尾に `0` が付く（`"60980"`）**。
フィルタ条件はこちらに修正が必要。

同日の1116件の中にリクルートHDの提出書類が3件ヒット：

```json
{
  "docID": "S100YDHL",
  "edinetCode": "E07801",
  "secCode": "60980",
  "filerName": "株式会社リクルートホールディングス",
  "ordinanceCode": "010",
  "formCode": "030000",
  "docTypeCode": "120",
  "periodStart": "2025-04-01",
  "periodEnd": "2026-03-31",
  "docDescription": "有価証券報告書－第66期(2025/04/01－2026/03/31)",
  "csvFlag": "1"
}
```

→ Step2で「予想」としていた `docTypeCode == "120"` / `ordinanceCode == "010"` / `formCode == "030000"` は
すべて実データと一致することを確認できた（別紙1の値をここで確定できたので、Step2以降の「要実機確認」は解消）。
`edinetCode = "E07801"` も確定。

同日には他に `docTypeCode: "235"`（内部統制報告書）、`"135"`（確認書）もヒットしている。
有価証券報告書と同時に提出されることが多いので、フィルタでは必ず `docTypeCode == "120"` を明示すること。

## Step 2：5期分の「有価証券報告書」の提出日を特定する ✅実機検証済み

リクルートHDは3月決算なので、有価証券報告書は毎年おおむね**6月中旬〜下旬**に提出される。
書類一覧APIは日付指定必須なので、直近5年分それぞれについて、狙った日付（前年の実績日）を起点に
前後に日付をずらしながら1日ずつ叩き、以下の条件に一致する書類を探す。

```
GET https://api.edinet-fsa.go.jp/api/v2/documents.json
    ?date=2025-06-XX
    &type=2
    &Subscription-Key={APIキー}
```

絞り込み条件（レスポンスの `results[]` 内）：

- `secCode == "60980"`（Step1で判明した5桁表記）
- `docTypeCode == "120"`（有価証券報告書。訂正版は `"130"` なので除外）
- `ordinanceCode == "010"`（企業内容等の開示に関する内閣府令）
- `formCode == "030000"`

一致した行の `docID`（書類管理番号）を5期分（5件）集める。

### 実機検証結果（2026-07-19実施）

6月20日前後を起点に±25日の範囲で総当たり探索するPythonスクリプトを書いて実行（各リクエスト間0.6秒sleepで429回避）。
5期分すべての `docID` を特定できた。**提出日は年によって数日〜1週間程度前後にずれる**ため、
「毎年同じ日」決め打ちではヒットしないことを確認（例：2023年は6/22ではなく6/27に提出）。

| 会計期間（第何期） | docID | 提出日 |
|---|---|---|
| 2021-04-01〜2022-03-31（第62期） | `S100O9QT` | 2022-06-22 |
| 2022-04-01〜2023-03-31（第63期） | `S100R3LU` | 2023-06-27 |
| 2023-04-01〜2024-03-31（第64期） | `S100TNM6` | 2024-06-21 |
| 2024-04-01〜2025-03-31（第65期） | `S100VZG5` | 2025-06-23 |
| 2025-04-01〜2026-03-31（第66期） | `S100YDHL` | 2026-06-19 |

さらに最も古い年（`S100O9QT`, 2022年提出分）を実際に②書類取得API（`type=5`）でCSV取得し、
Step3で確定した要素ID（`RevenueIFRSSummaryOfBusinessResults` / `ProfitLossAttributableToOwnersOfParentIFRSSummaryOfBusinessResults` /
`TotalAssetsIFRSSummaryOfBusinessResults` / `OperatingProfitLossIFRS` / `LiabilitiesIFRS`）が
**同じ要素IDで5件とも一致して取得できる**ことを確認済み。5年分すべて同じパースロジックで扱える。

**効率化のTIPS**：日付総当たりは429を避けるため必ずsleepを挟むこと。EDINETの関覧サイト（ブラウザ）で
先に「リクルートホールディングス」を書類簡単検索し、5期分の提出日を目視で確認してから
その日付だけを①APIに投げれば、総当たりより高速。

## Step 3：書類（財務データ）を取得する

設計書（`docs/design/cycle1_design.md`）では「XBRLパース：EDINETの財務サマリーCSVを使用」と
決まっているので、`type=5`（CSV）で取得するのが本命。

```
GET https://api.edinet-fsa.go.jp/api/v2/documents/{docID}
    ?type=5
    &Subscription-Key={APIキー}
```

| type値 | 内容 | ファイル形式 |
|---|---|---|
| 1 | 提出本文書＋監査報告書（XBRL含む） | ZIP |
| 2 | PDF | PDF |
| 3 | 代替書面・添付文書 | ZIP |
| 4 | 英文ファイル | ZIP |
| 5 | **CSV**（XBRLをCSVに変換したもの）← 今回はこれ | ZIP |

- ZIPを解凍すると `XBRL_TO_CSV/` フォルダの中に複数CSVが入っている
- 取得前に書類一覧APIのレスポンスの `csvFlag` が `"1"` であることを確認（`"0"`だとCSVなし＝取得不可）

これを5期分（5回）実行すれば、リクルートの5年分の財務データCSVが揃う。

### 実機検証結果（2026-07-19実施）✅

`docID: S100YDHL` に対して実際に `type=5` を実行し、ZIPを解凍。3つのCSVが得られた：

```
jpaud-aai-cc-001_E07801-000_2026-03-31_01_2026-06-19.csv   … 監査人の意見表明区分
jpaud-aar-cn-001_E07801-000_2026-03-31_01_2026-06-19.csv   … 監査報告書
jpcrp030000-asr-001_E07801-000_2026-03-31_01_2026-06-19.csv … 提出本文書（有価証券報告書本体）★これが本命
```

`jpcrp030000-asr-001_*.csv` の中身：
- 文字コードは **UTF-16LE**、タブ区切り、CRLF改行（`iconv -f UTF-16LE -t UTF-8` で変換して確認）
- ヘッダ：`要素ID / 項目名 / コンテキストID / 相対年度 / 連結・個別 / 期間・時点 / ユニットID / 単位 / 値`
- **1ファイルの中に「経営指標等」として5期分（当期・前期・前々期・三期前・四期前）がまとめて入っている**ので、
  日付を変えて5回ダウンロードしなくても、直近の1件を取得するだけで5期分の主要指標は揃う
  （※詳細な注記や個別の期の全項目まで欲しい場合は従来どおり5期分の書類を集める）

設計書で欲しい5指標に対応する要素IDは実データで以下のとおり確認できた：

| 設計書の指標 | 要素ID | 値の場所 | 備考 |
|---|---|---|---|
| 売上高 | `jpcrp_cor:RevenueIFRSSummaryOfBusinessResults` | 経営指標等（5期分あり） | 相対年度：`CurrentYearDuration`〜`Prior4YearDuration` |
| 営業利益 | `jpigp_cor:OperatingProfitLossIFRS` | 連結損益計算書本体 | **経営指標等サマリーには含まれない**。書類ごと（＝期ごと）に本体から1件ずつ拾う必要あり |
| 純利益 | `jpcrp_cor:ProfitLossAttributableToOwnersOfParentIFRSSummaryOfBusinessResults`（親会社帰属分） | 経営指標等（5期分あり） | 連結全体の`ProfitLossIFRSSummaryOfBusinessResults`もあるが、設計書の「純利益」は親会社帰属分を使うのが一般的 |
| 総資産 | `jpcrp_cor:TotalAssetsIFRSSummaryOfBusinessResults` | 経営指標等（5期分あり） | 相対年度：`CurrentYearInstant`〜`Prior4YearInstant` |
| 負債合計 | `jpigp_cor:LiabilitiesIFRS` | 連結財政状態計算書本体 | **経営指標等サマリーには含まれない**。営業利益と同様、書類ごとに1件ずつ拾う必要あり |

**重要な設計上の注意**：
売上高・純利益・総資産は「経営指標等」サマリーに5期分まとまっているので直近1件のCSVだけで済むが、
**営業利益と負債合計はサマリーに存在せず、各期の本体（連結損益計算書／連結財政状態計算書、`CurrentYearDuration`/`CurrentYearInstant`の値のみ）からしか取れない**。
そのため結局5期分の書類を個別に取得する必要がある（Step2の日付ループが必要）。
`xbrl_parser.py` を実装する際は、要素IDでの完全一致マッチが必須（`jpcrp_cor:...SummaryOfBusinessResults`系と`jpigp_cor:...IFRS`系の2系統を組み合わせてパースする）。

## 検証時の注意点

- **429 (Too Many Requests)**：短時間に大量アクセスすると発生。日付ループで叩く場合は
  リクエスト間に数百ms〜1秒程度のsleepを入れる。
- **閲覧期間**：有価証券報告書は縦覧5年＋延長5年＝最大10年閲覧可能なので、直近5年分は問題なく取得できる。
- **書類管理番号(docID)は書類一覧APIでしか手に入らない**ため、Step2を必ず先に行うこと（一発で書類取得APIだけ叩くことはできない）。
- クロスドメイン通信非対応 → ブラウザJSからは叩けない。バックエンド（FastAPI）経由でのみ実行する設計は妥当。

---

## 動作検証のミニマム手順（設計フェーズでの確認用）

1. APIキーを取得
2. Step1のリクエストを1回叩き、レスポンスJSONの中に `secCode: "6098"` の行があるか確認
3. その行の `edinetCode` と `docDescription` をメモ
4. 見つかった日付が有価証券報告書の提出日でなければ、日付を変えて2〜3を繰り返す
5. 有価証券報告書が見つかったら `docID` を使ってStep3（type=5）を1回実行し、CSVが正しくダウンロードできるか確認

ここまで通れば「APIでリクルートの5年分データが取得可能」という設計フェーズの検証は完了。
