## 5. 要件トレーサビリティチェック（FRの箇条書き単位で設計と突き合わせ）

- [x] FR-42「モジュールレベルのカウンタを追加」→ 設計§1で`_request_count`として一致
- [x] FR-42「`_get`が呼ばれるたびにインクリメント」→ 設計§1で一致
- [x] FR-42「読み取り・リセット関数」→ 設計§1で`get_request_count`/`reset_request_count`
  として一致
- [x] FR-43「業種・会計基準に偏りが出ないよう分散」→ **一部具体化**：要件は
  「業種・会計基準」の両方を挙げていたが、サイクル6時点で会計基準が判明している
  企業は14社のみ（要件定義書にも明記済み）と少なすぎるため、設計では実質的に
  「業種のみで分散」とし、要件文の「業種のみで分散させた未取得企業を中心に選定する」
  という但し書きと整合する形にした。矛盾ではなく要件定義書の記述をそのまま反映
- [x] FR-43「既存のダウンロードAPIを使う」→ **設計で具体化**：要件は
  「`POST /api/download`・`GET /api/download/status`」とAPI経由を想定した書き方だったが、
  設計では`run_download_job`を直接呼び出す方式にした（ロジックは完全に同一、
  HTTPサーバー起動が不要になる分シンプル）。要件の意図（既存ロジックをそのまま使う、
  新規のダウンロードロジックを作らない）は満たしているため、要件定義書の修正は不要と
  判断したが、「既存のAPIをそのまま使う」という文言は「既存のダウンロードロジックを
  そのまま使う」の方が正確だった
- [x] FR-43「所要時間・リクエスト回数・成功/失敗を記録」→ 設計§2で一致
- [x] FR-43「失敗しやすい企業も意図的に含める」→ 設計§2「サンプル選定方針」で
  「小規模企業を2〜3社手動で加える」として一致（具体的な企業選定は実装時に確定）
- [x] FR-44「実測値から全社展開時の所要時間を試算」→ 設計§3のドキュメントひな形に
  「全社展開時の試算」の節として一致
- [x] FR-44「フェーズ3への課題を記録」→ 設計§3に「フェーズ3への申し送り事項」として一致
- [x] NFR-16「EDINETへの負荷配慮」→ 設計§2でサンプル数を20社程度に限定、
  レート制限機構は変更しない設計として一致
- [x] NFR-17「取得したデータはDBに保存される（意図した副作用）」→ 設計は
  `run_download_job`をそのまま使うため、既存の`_upsert_facts`等の保存処理がそのまま
  動作する。設計書に明記はしていないが、動作としては要件の想定通り

## 判定：実装フェーズへ進んで問題なし

要件の全箇条書きを設計と突き合わせ、抜け漏れ・矛盾は見つからなかった。FR-43の
「既存のAPIをそのまま使う」という文言と設計の「関数を直接呼び出す」という選択は、
要件の意図（既存ロジックの再利用）を満たす具体化であり、要件定義書の修正は不要と
判断した。実装順：FR-42（`edinet_client.py`のカウンタ・`to_company_code`共通化）→
FR-43（`verify_batch_timing.py`・実行）→ FR-44（見積もりドキュメント作成）。
