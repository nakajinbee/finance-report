# 設計 セルフレビュールール

設計完了後、開発フェーズに進む前に必ず実施すること。
レビュー結果は `cycle{N}_design_review.md` に記録する。

---

## 1. 機能面：要件を満たす設計になっているか

- [ ] 要件定義のすべての機能要件（FR-xx）に対応する設計が存在するか
- [ ] 各機能の入力・処理・出力がデータフロー図や API 設計に反映されているか
- [ ] エラー・例外ケース（API失敗・データなし・入力不正）の処理フローを設計しているか
- [ ] ユーザーの全操作（画面遷移・ボタン・フォーム）を画面設計に含めているか
- [ ] カスタマイズ要件（指標の追加・削除・保存）の実現手段を設計しているか

---

## 2. 性能面：非機能要件を満たす設計になっているか

- [ ] 応答時間の要件（検索3秒・データ表示10秒）を実現できる構成か
- [ ] EDINET API のレート制限を超えない設計になっているか
- [ ] 必要に応じてキャッシュ戦略を設計しているか（重複リクエストの抑制）
- [ ] XBRL ファイルのパース処理がボトルネックにならないか検討したか

---

## 3. セキュリティ・運用面

- [ ] APIキーをフロントエンドに露出させない設計になっているか
- [ ] ユーザー入力をサーバーサイドでバリデーションする設計か
- [ ] ローカルストレージに保存するデータの安全性を考慮しているか

---

## 4. 拡張性

- [ ] サイクル2の追加要件（複数年推移・複数企業比較など）を壊さずに追加できる構造か
- [ ] コンポーネント・モジュールの責務が分離されているか
