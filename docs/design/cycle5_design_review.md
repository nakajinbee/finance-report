## 5. 設計内容の検証（本サイクル固有の確認）

実装前に、設計で「対象」とした箇所が実際のコードと一致しているか`grep`で裏取りした。

- [x] `THEME`オブジェクトが本当に未使用か　→ `grep -rn "THEME" frontend/src`で
  定義ファイル自身以外にヒットしないことを確認した。削除して問題ない
- [x] `--color-brand-light`・`--color-tint`トークンの参照箇所を全て把握できているか　→
  `grep`で`Header.tsx`・`Footer.tsx`・`index.css`定義部の3箇所のみと確認した。FR-35の
  `Header`/`Footer`書き換えで参照が消えるため、トークン削除後にビルドエラーが出る
  参照漏れはない
- [x] `bg-sky-50`（サイクル4の暫定ハードコード）の参照箇所を網羅できているか　→
  `grep`で`MetricSelector.tsx`・`RatioToggle.tsx`の2箇所のみと確認した
- [x] FR-36の「入力欄」対象リストに、フォーカス不可の要素を誤って含めていないか　→
  レビュー中に`DownloadPage.tsx`188行目の「選択中：〇〇」表示`<div>`（`border-gray-300`が
  付いているが`<input>`ではない）を対象に含めかけていたことに気づき、設計を修正して
  明示的に対象外とした
- [x] ネイティブのradio/checkboxの扱いが設計から漏れていないか　→
  `DownloadPage.tsx`の取得期間選択（radio×2）はdesign_guideline.mdに規定がないことに
  気づき、「対象外・ブラウザ標準のまま」と設計に明記した（未定義のまま実装に進むと、
  実装者ごとに対応が割れる曖昧さが残るため）
- [x] `tabular-nums`を`body`に一括適用する設計が、Tailwind Preflightの`font: inherit`
  リセットと衝突しないか　→ Preflightは`button, input, select, ...`に`font: inherit`を
  適用しており、`inherit`は初期値へのリセットではなく親（`body`）からの継承を意味するため、
  `font-variant-numeric: tabular-nums`はフォーム部品にも伝播する。個別要素へのクラス
  追加漏れが起きない設計として妥当

## 6. 要件トレーサビリティチェック（FRの箇条書き単位で設計と突き合わせ）

「要件を満たす設計になっているか」を1節でチェック済みだが、それはFR単位（見出し単位）の
対応確認であり、各FR内の**箇条書き1つずつ**が設計に反映されているかまでは検証していなかった。
改めて[cycle5_requirements.md](../requirements/cycle5_requirements.md)の箇条書きを1件ずつ
設計書と突き合わせた。

- [x] FR-34「`--color-brand`等3トークンを定義」→ 設計§1で値まで一致
- [x] FR-34「`--color-brand-light`・`--color-tint`削除」→ 設計§1で一致
- [x] FR-34「`THEME`も同じ値に更新」→ **不一致を発見**：設計は「未使用のため更新ではなく
  削除」という判断をしていたが、要件文はそれ以前に書いた「更新する」のままだった。
  設計判断（削除）自体は`grep`による裏付けがあり妥当なため、要件定義書側を実態に合わせて
  修正した（「更新する」→「未使用コードのため削除する」）
- [x] FR-34「`CHART_COLORS`等は変更しない」→ 設計§1で一致
- [x] FR-35の3項目（中立色統一・Panel背景・Header/Footer）→ 設計§2で一致
- [x] FR-36「フォーカス時の見た目」→ 設計§3で一致
- [x] FR-36「disabled状態に`bg-gray-100`・`cursor-not-allowed`」→ **不一致を発見**：
  設計は既存コードに`disabled:bg-gray-100`があることだけを確認して「そのままでよい」と
  結論していたが、要件が求めるもう一方の`cursor-not-allowed`が既存コードになく、
  見落としていた。設計書に`disabled:cursor-not-allowed`を追加するdiffを追記した
- [x] FR-36「トグルの選択色」→ 設計§3で一致
- [x] FR-36「カードホバー」→ 設計§3で一致（既存実装が要件を満たすことの確認として記載）
- [x] FR-37の2項目（`tabular-nums`・`CartesianGrid`）→ 設計§4で一致
- [x] FR-38の4項目（エラー色・バッジ・アイコン・ローディング）→ 設計§5で一致
- [x] NFR-11・NFR-12（影響範囲・既存機能への影響）→ 本サイクルの変更はいずれもスタイルの
  みで、機能的な挙動を変えるものがないことを設計内容から確認

## 判定：実装フェーズへ進んで問題なし

レビュー中に計4件（フォーカス不可の`<div>`を対象に含めかけていた点、radio/checkboxの
scope外明記漏れ、FR-34の「THEME更新」記述と設計の「削除」判断の不一致、FR-36の
`cursor-not-allowed`見落とし）を発見し、要件定義書・設計書の両方を修正済み。修正後、
要件の全箇条書きが設計に反映されていることを確認し、抜け漏れ・矛盾は見つからなかった。
実装順：FR-34 → FR-35 → FR-36 → FR-37（FR-38はコード変更なしの確認のみ）。
