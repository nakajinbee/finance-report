## 1. FR-52：データ量見積もりの実測値更新

### 実測方法

`financials.db`に対し、以下のクエリを実行して基礎データを取得する（実装フェーズで
そのまま実行し、結果を`cycle10_data_estimate.md`に記録する）。

```sql
-- テーブル・インデックスごとの実サイズ（sqlite dbstat仮想テーブル）
SELECT name, SUM(pgsize) FROM dbstat GROUP BY name ORDER BY 2 DESC;

-- 1書類あたりの平均facts行数
SELECT COUNT(*) AS total_facts, COUNT(DISTINCT doc_id) AS total_docs
FROM facts;

-- 「初回バックフィル時の1社あたり平均書類数」＝複数書類を取得済みの
-- 企業（サイクル7/8の技術検証で全期間ダウンロード済みの企業）のみを対象に算出
SELECT company_code, COUNT(DISTINCT doc_id) AS docs
FROM facts GROUP BY company_code HAVING docs >= 2;
```

事前確認済みの実測値（要件定義セルフレビュー時点、2026-07-23時点のDB）：

| 項目 | 値 |
|---|---|
| `facts`テーブル＋関連インデックスの実サイズ | 208,904,192 bytes（約199.2 MB） |
| `facts`総行数 | 482,692行 |
| `facts`に含まれる書類数（distinct doc_id） | 515件 |
| **1書類あたり平均facts行数・サイズ（インデックス込み）** | **約937行・約405,639 bytes（約396 KB）** |
| `documents`テーブル＋関連インデックスの実サイズ | 7,176,192 bytes（約6.84 MB） |
| `documents`総行数（過去10年分バックフィル済み、全社分完了） | 41,567行 |
| 全期間ダウンロード済み企業（docs≧2）の1社あたり平均書類数 | 約9.3件（min 2・max 12） |

### 全社換算の推定（初回バックフィル）

- 対象企業数：3,829社（`companies`登録済み全社）
- 推定書類数：3,829社 × 約9.3件/社 ≈ **約35,600件**
- 推定`facts`サイズ：35,600件 × 約396KB/件 ≈ **約14.1 GB**
  （サイクル7の見積もり「約14.6GB」とほぼ一致。サンプルが33社→83社に増えても
  同水準の見積もりに収束したことで、この規模感の信頼性が上がったと判断する）
- `documents`テーブルは既に全社分の10年分バックフィルが完了済み（41,567行・
  約6.84MB）のため、初回分の追加増加はない

### 定常運用時の年間増加量（IDEA-01原文の「1年経過ごとの増加量」に対応）

初回バックフィル完了後、各社は年1件（有価証券報告書）〜2件（＋半期報告書）の
新規書類を提出する。

- `facts`：3,829社 × 1〜2件/年 × 約396KB/件 ≈ **年間約1.5GB〜3.0GB**
- `documents`：3,829社 × 1〜2件/年 × 約173bytes/件（`documents`側の1件あたり
  実サイズ：7,176,192 / 41,567 ≈ 173bytes） ≈ 年間1MB未満（無視できる規模）

## 2. FR-53：並行アクセスパターンの整理

### 現状のSQLite設定の実測

```python
# 実行結果（financials.dbに対して直接確認）
PRAGMA journal_mode;   # → 'delete'（デフォルトのロールバックジャーナル。WALではない）
PRAGMA busy_timeout;   # → 5000（ミリ秒。SQLAlchemyのデフォルト値がそのまま有効）
```

`journal_mode='delete'`はコミット時に短時間の排他ロックを取る方式で、WALモード
（`journal_mode='wal'`）と比べると「書き込み中の読み取り」の並行性は低い。ただし
`busy_timeout=5000`が既に設定されているため、ロック競合が発生してもエラーには
ならず、最大5秒間はリトライ待機してからアクセスする。

### 実際の書き込みパターン（サイクル9の実装を前提に整理）

| 処理 | 書き込み頻度・粒度 | 備考 |
|---|---|---|
| `document_list_ingestion.ingest_document_list_for_date` | 1日1回、1トランザクションで`documents`へupsertしまとめてcommit | 将来の日次自動化後、1日1回・数秒未満で完了する短時間の書き込みになる見込み（EDINETの1日あたり対象書類数は数十件程度） |
| `document_body_ingestion.ingest_document_body` | 1書類ごとに1コミット（`document_body_ingestion.py`の`session.commit()`は書類単位） | 意図的に書類単位でコミットしており、1回のロック保持時間は短い（設計時点からの既存方針） |
| `routers/edinet.py`の個別ダウンロード | ユーザー操作時のみ、期間ごとに1コミット | 頻度は低い（ユーザーが明示的に実行した時のみ） |
| FastAPIの読み取り（`routers/companies.py`等） | リクエストの都度、読み取りのみ | 書き込みとは独立したセッション |

### 評価

- 書き込みは「1日1回のバッチ（短時間）」＋「書類単位でコミットする本体取り込み
  （さらに短時間）」＋「ユーザー操作時のみの個別ダウンロード」であり、いずれも
  長時間のトランザクションを保持しない設計に既になっている
- 読み取り（画面表示のAPI）は書き込みと同一プロセス内で競合しても、
  `busy_timeout=5000`により待機後にリトライされるため、エラーにはならない
  （応答が数秒遅延する可能性はあるが、致命的な失敗ではない）
- **SQLiteが不向きになるのは、複数のアプリケーションサーバー（別マシン・別プロセス群）
  から同時にDBファイルへ書き込む「水平スケーリング」構成になった場合**（SQLiteは
  単一ファイルへの組み込み型DBのため、複数マシンからの同時書き込みという構成自体が
  そもそも成立しない）。現状もサイクル9の設計も、単一プロセス（1台のサーバー）での
  運用を前提にしており、複数サーバーでの水平スケーリング計画は存在しない
- 上記より、現状およびサイクル9で実装した書き込みパターンの範囲では、SQLiteの
  同時書き込み制約が実運用上の問題になる可能性は低いと判断する

## 3. FR-54：判断の記録

### 記録先

新規ファイル`docs/development/cycle10_db_migration_decision.md`を作成する
（`cycle7_batch_timing_estimate.md`は「サイクル7〜8の技術検証」という文脈の
ドキュメントであり、フェーズ6の最終判断は独立したファイルに切り出した方が
将来参照しやすいと判断。`cycle7_batch_timing_estimate.md`側には本ファイルへの
参照リンクのみ追記する）。

### 記録する内容

- FR-52の実測値・全社換算推定（初回バックフィル約14.1GB、年間増加量約1.5〜3.0GB）
- FR-53の並行アクセスパターン評価
- 結論：**SQLiteを継続する**。理由：(1)データ量は約14GB規模で技術的な上限に対し
  十分小さい、(2)現状・サイクル9実装後の書き込みパターンはいずれも短時間・低頻度で
  `busy_timeout`により実運用上の問題になりにくい、(3)水平スケーリング（複数サーバー
  からの同時書き込み）の計画が現時点で存在しない、以上3点から移行の必要性がない
- 判断が覆る条件（将来の再検討トリガー）を明記する：複数サーバーでの運用が
  具体的に計画された場合、または実測のロック競合・書き込み遅延が実際に問題として
  報告された場合は、その時点で移行を再検討する

### `docs/ideas/IDEA-01_db_batch_ingestion.md`の更新

- フェーズ6の状態を「完了（サイクル10、判断：SQLite継続）」に更新し、
  `cycle10_db_migration_decision.md`へのリンクを追記する
