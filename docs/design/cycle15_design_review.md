# サイクル15 設計セルフレビュー

対象：[cycle15_requirements.md](../requirements/cycle15_requirements.md)。設計成果物は
定義書群（`docs/design/screen/`・`docs/design/api/`）を直接更新したもの
（`cycleN_design.md`は作らない。cycle-workflow参照）。

## 1. 機能面

- FR-65〜69すべてに対応する画面定義書（SCR-005・SCR-006・SCR-007・SCR-003更新）・
  API定義書（API-COM-007）が存在する
- エラー・例外ケース（API失敗・データなし・入力不正）：SCR-005（絞り込み結果0件、
  データなし）・SCR-006（一部失敗・空状態）・SCR-007（結果0件・エラー）すべてに
  画面状態表を用意した
- ユーザーの全操作（検索・業種絞り込み・並び順・企業選択・除外・比較する・
  グラフ企業名クリック等）を各画面の機能定義・項目一覧に含めた

## 2. 性能面

- ランキングAPI（API-COM-007）は全社対象時に約3,800社分のループ処理になる。
  設計時点ではまだ実装前のため実測できないが、**実装セルフレビューで必ず
  実際の応答時間を計測する**ことをNFR-15・本レビューの両方に明記する
  （未実測のままフロント実装に進まない）
- 比較結果画面（SCR-006）は対象企業数が少数（数社〜十数社想定）である前提を
  明記した。既存API（API-COM-002/003/005）の並行呼び出しで対応する設計とし、
  新規の集計APIは作らない（YAGNI）

## 3. セキュリティ・運用面

- 比較企業選択画面（SCR-005）の選択状態はクライアント側の一時的な状態のみで
  保持し、バックエンドには何も送信・保存しない（機微情報を含まないため問題なし）
- SCR-006の`codes`クエリパラメータはバックエンドAPIへの入力ではなく、フロント側で
  証券コードごとに既存APIを呼び出す際のキーとして使うのみ（バックエンド側の
  入力バリデーションは既存のAPI-COM-002/003/005のパスパラメータ検証がそのまま働く）

## 4. 拡張性

- ランキングAPIの`metric`は列挙型で厳格に定義しており、新しい指標を追加する際は
  `metric_mappings.py`とAPI定義書のenumの両方を更新すればよい構造になっている
- 比較結果画面のカテゴリ構成をSCR-003と共通化したことで、将来SCR-003に指標が
  追加された場合も同じ分類にそのまま追従できる

### NFR-15実測結果（実装後に追記）

`curl`で`GET /api/companies/ranking?metric=revenue`（全社対象、業種フィルタなし）を
実行し応答時間を計測した。全社（3,829社）をループする実装だが、2026-07-24時点で
実際にDBへ取り込み済みの企業は61社のみのため、今回の実測値はまだ「全社に
データが揃った将来」の負荷を表していない点に注意。

- 全社対象（`metric=revenue`）：約4.3秒
- 全社対象（`metric=roe`）：約4.4秒
- 業種絞り込みあり：対象企業数に比例して短縮

現時点のデータ量では実用上問題ない時間だが、将来的に全上場企業（約3,800社）に
データが揃った場合、単純比例だと数分規模になりうる。数秒〜数十秒程度に収まるかは
実データが揃った時点で再計測が必要。`cycleX_backlog.md`に「全社データ投入後の
ランキングAPI応答時間の再検証・必要なら集計済みキャッシュテーブルの検討」として
記録する。
