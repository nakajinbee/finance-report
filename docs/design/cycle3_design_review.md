## 1. 機能面

- [x] 要件定義のすべての機能要件（FR-17・18・19）に対応する設計が存在するか　→
  それぞれ独立した節で設計済み。FR-20は要件定義で対象外と明記済みのため設計不要
- [x] 各機能の入力・処理・出力がデータフロー図やAPI設計に反映されているか　→
  FR-17に入力・処理・出力の表、FR-18・19に処理フロー（疑似コード）を記載済み
- [x] エラー・例外ケース（API失敗・データなし・入力不正）の処理フローを設計しているか　→
  「候補が1つも見つからない場合はデータなし」「未知の決算日表記は既存動作を踏襲」を明記済み
- [x] ユーザーの全操作（画面遷移・ボタン・フォーム）を画面設計に含めているか　→
  本サイクルは画面変更を伴わないため対象外（ヘッダーに明記済み）
- [x] カスタマイズ要件（指標の追加・削除・保存）の実現手段を設計しているか　→
  本サイクルはカスタマイズ要件を含まないため対象外

## 2. 性能面

- [x] 応答時間の要件を実現できる構成か　→ 候補リスト走査のコストは無視できるオーダーと
  設計書に明記
- [x] EDINET API のレート制限を超えない設計になっているか　→ 本サイクルはEDINET通信部分を
  変更しないため影響なし
- [x] 必要に応じてキャッシュ戦略を設計しているか　→ 新規の外部呼び出しを追加しないため
  キャッシュ戦略は不要（対象外）
- [x] XBRL ファイルのパース処理がボトルネックにならないか　→ `xbrl_parser.py`のパース処理
  自体は本サイクルで変更しない。変更対象はパース後のfacts索引・照合ロジックのみ

## 3. セキュリティ・運用面

- [x] APIキーをフロントエンドに露出させない設計になっているか　→ 変更なし（対象外）
- [x] ユーザー入力をサーバーサイドでバリデーションする設計か　→ 新規のユーザー入力を
  追加しないため対象外
- [x] ローカルストレージに保存するデータの安全性を考慮しているか　→ フロントエンド変更が
  ないため対象外

## 4. 拡張性

- [x] 既存要件を壊さずに追加できる構造か　→ 候補リスト方式・共通化されたコンテキスト
  フォールバックにより、新しい企業・新しいタグが見つかっても`metric_mappings.py`への
  追記のみで対応できる設計にした
- [x] コンポーネント・モジュールの責務が分離されているか　→ タグの候補管理（データ）は
  `metric_mappings.py`、照合ロジック（処理）は`routers/companies.py`、決算日パースは
  `edinet_client.py`と、サイクル2までの責務分担を維持している

## ドメイン知識との整合性確認（今回のレビューで発見した不整合）

設計書のFR-18節を[docs/domain/xbrl_tagging_variability.md](../domain/xbrl_tagging_variability.md)
と突き合わせたところ、**両方の文書に同一の誤りがあった**：

> 「連結・非連結どちらのコンテキストで見つかったかは、`facts.consolidated_or_individual`列
> （サイクル2実装済み）で判別可能」

実データ（大本組）を再確認したところ、`consolidated_or_individual`列（CSVの「連結・個別」列を
そのまま保存したもの）は`_NonConsolidatedMember`コンテキストの行でも`"個別"`・`"その他"`など
値が一定しておらず、信頼できる判別材料ではないことが判明した。連結・非連結の判別は
**コンテキストIDの`_NonConsolidatedMember`サフィックスの有無**で行うべきであり、
`consolidated_or_individual`列に依存すべきではない。

この誤りは`cycle3_design.md`・`docs/domain/xbrl_tagging_variability.md`の両方を
本レビューで修正済み。

---

## 判定：実装フェーズへ進んで問題なし

上記1点（`consolidated_or_individual`列への誤った依存）を除き、抜け漏れ・矛盾は
見つからなかった。修正済みのため実装に進む。
