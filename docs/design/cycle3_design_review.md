## ドメイン知識との整合性確認（今回のレビューで発見した不整合）

設計書のFR-18節を[docs/domain/xbrl_tagging_variability.md](../domain/xbrl_tagging_variability.md)
と突き合わせたところ、**両方の文書に同一の誤りがあった**：

> 「連結・非連結どちらのコンテキストで見つかったかは、`facts.consolidated_or_individual`列
> （サイクル2実装済み）で判別可能」

実データ（大本組）を再確認したところ、`consolidated_or_individual`列（CSVの「連結・個別」列を
そのまま保存したもの）は`_NonConsolidatedMember`コンテキストの行でも`"個別"`・`"その他"`など
値が一定しておらず、信頼できる判別材料ではないことが判明した。連結・非連結の判別は
**コンテキストIDの`_NonConsolidatedMember`サフィックスの有無**で行うべきであり、
`consolidated_or_individual`列に依存すべきではない。

この誤りは`cycle3_design.md`・`docs/domain/xbrl_tagging_variability.md`の両方を
本レビューで修正済み。

## 追記（2026-07-22）：FR-23〜26（財務分析指標の拡充）のレビュー

### 1. 機能面
- [x] 要件定義のFR-23〜26すべてに対応する設計があるか　→ 「6. FR-23〜26」節に
  データ構造・処理フロー・スキーマ・フロントエンド設計を網羅
- [x] 開示値・計算値の優先順位（ユーザー承認済み）が処理フローに反映されているか　→
  `equity_ratio = disclosed.get("equity_ratio") or _safe_div(...)`のように反映済み
- [x] エラー・例外ケース（ゼロ除算・データなし）を設計しているか　→ `_safe_div`で対応
- [x] US GAAP企業が全指標データなしになることを明記しているか　→ 明記済み

### 2〜4. 性能・セキュリティ・拡張性
- [x] 性能：既存の`_index_facts_by_period`・`_lookup_metric`を再利用するため、
  追加コストは候補走査分のみで無視できるオーダー
- [x] セキュリティ：新規GETエンドポイントのみ追加、書き込み系APIではないため
  入力バリデーションの必要性は`from_year`/`to_year`のみ（既存の`_validate_year_range`を再利用）
- [x] 拡張性：`DISCLOSED_RATIOS`・`BALANCE_SHEET_ITEMS`とも既存のFIVE_METRICSと同じ
  「会計基準→指標名→候補ローカル名リスト」という形式で統一しているため、新しい指標の
  追加は既存パターンの踏襲で対応できる

### ドメイン知識との整合性
`docs/domain/xbrl_tagging_variability.md` 6節（開示値と計算値の二重存在）と、
設計書の「開示値優先、なければ計算値で補完」ロジックが一致していることを確認した。

### 発見事項
- `equity`（自己資本）をグループA（equity_ratio計算の補完用）とグループC
  （fixed_ratio・current_ratioの計算用）の両方で使う設計にしたことで、
  `BALANCE_SHEET_ITEMS`に1箇所だけ定義すればよく、重複が生じないことを確認した
  （設計時点で意図した通り）
