# サイクル12 要件定義書

対象：[IDEA-13](../ideas/IDEA-13_use_case_design.md)（ユースケース設計）

**前提確認**：本サイクルはサイクル11同様、通常の機能追加とは性質が異なる企画
フェーズ（[cycle-workflow](../../.claude/skills/cycle-workflow/SKILL.md)のStep 0
相当）。成果物はコード・画面の変更ではなく、`docs/product/concept.md`（サイクル11で
確定）を土台にユーザーとの対話で洗い出したユースケース一覧のドキュメント化である。

## プロダクト概要（サイクル11までの状況）

サイクル11でアプリのビジネスコンセプト（想定利用者・提供価値・開発方針・
事業化スタンス）を確定した。ユーザーから「今の画面は用途を伴わずただ数値を表示
するだけの画面としてポツンと存在している」という課題認識が示され、目的・利用
シーンを構造化してユースケースを固め、それを実現するための画面・API・バッチを
設計する、という進め方に変更する方針が示された（既存の画面の中には、ユースケース
策定の結果、不要と判明するものも出てくる前提）。

## サイクル12の目標

コンセプトをもとに「やりたいことの整理」を行い、大項目・中項目・小項目
（最大3階層）でユースケースを構造化し、`docs/product/use_cases.md`として確定する。
以降のサイクルは、ここで洗い出したユースケース（主に中項目単位）を対象に、
画面フロー→画面→API→バッチの順で設計・実装していく。

## 決定した内容：ユースケース構造（大項目3・中項目13）

ユーザーとの対話（複数回のAskUserQuestion・自由記述でのフィードバック）を通じて
以下の3段階の精査を行った：

1. 大項目の洗い出し：投資判断支援・会計学習・データ蓄積/時系列分析という
   コンセプトの提供価値に対応する3つの軸が確定
2. 中項目の洗い出し：各大項目に対し、具体的な利用シーンを複数提示し、
   選択・追加・統合を経て確定（例：会計学習の「業種・会計基準ごとの違いを学ぶ」は
   独立した中項目ではなく「分析時の注意点」に格下げ、「学習モード」は新規中項目
   として追加）
3. 小項目の精査：中項目ごとに、開発者視点の機能（生データ確認等）を除外し、
   曖昧な表現（「シェアを見る」等）を具体化する調整を行った

最終的なユースケース一覧は`docs/product/use_cases.md`に記録する（内容はFR-56参照）。

## FR-56：ユースケース一覧のドキュメント化

- `docs/product/use_cases.md`（新規）に、大項目3・中項目13・各中項目の小項目を
  記録する
- 各ユースケースについて、「実現済み（既存機能で対応可能）」「未実現（新規に
  画面・API・バッチが必要）」の状態を可能な範囲で注記し、以降のサイクル選定の
  参考にする
- `docs/ideas/IDEA-13_use_case_design.md`の状態を「完了」に更新する
- `docs/INDEX.md`に`docs/product/use_cases.md`へのリンクを追加する

## スコープ外（次サイクル以降）

- 各ユースケースを実現するための画面フロー・画面・API・バッチの設計（サイクル13
  以降、ユースケース単位で順次実施）
- IDEA-10（レポート表示の再設計）：ユースケースが確定した後、既存画面の要否判断・
  再設計を行う
- 大項目3-5（独自の理論株価算出・割安株スクリーニング）は、他のユースケースより
  技術的難度が高く、蓄積データの蓄積量にも依存するため、着手時期は他のユースケースの
  実装状況を見て個別に判断する
