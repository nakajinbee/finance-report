# サイクル1 要件定義 セルフレビュー結果

レビュー基準：self_review_rule.md

## 1. 抜け漏れチェック

### ユーザーアクション網羅
- [x] ユーザーが行う操作をすべて洗い出せているか
- [ ] 各操作の「入力・処理・出力」が定義されているか　→ **不足**
- [ ] 操作の結果として起こりうる全状態（成功・失敗・空）を定義しているか　→ **不足**

### 機能の網羅
- [x] ユーザーの「やりたいこと」に対応する機能がすべて書かれているか
- [x] 機能間の依存・順序に矛盾はないか
- [ ] エラー・例外時の挙動を定義しているか　→ **未定義**
  - 検索0件の場合
  - EDINET API がエラーを返した場合
  - 対象指標の XBRL データが存在しない場合
  - ネットワークエラーの場合

### 非機能要件の網羅
- [x] 性能要件を定義しているか
- [ ] セキュリティ要件を定義しているか　→ **未定義**（APIキーをどこで管理するか）
- [x] 対応環境を定義しているか
- [ ] データの鮮度・更新タイミングをユーザーに伝える要件があるか　→ **未定義**
  - 「直近の有価証券報告書」がいつ提出されたものか画面に表示する要件が必要

### スコープの明確化
- [x] スコープ内・外を明確に分けているか
- [x] スコープ外を次サイクル候補として記録しているか

---

## 2. 実現可能性チェック

- [ ] EDINET API で要件の指標が実際に取得できるか確認したか　→ **要確認**
  - 売上高・営業利益・純利益・総資産・負債は主要科目なので取得可能と推測するが、XBRL タグ名の確認が必要
  - 全企業で同一タグが使われているとは限らない（企業固有タグの存在）
- [ ] XBRL の勘定科目タグ一覧をユーザーに選択させる FR-02b の実現可能性　→ **要検討**
  - EDINET の XBRL タグは数百～数千存在し、全件リストは膨大になる可能性がある
  - カスタマイズ方式（全タグ選択 vs よく使う指標から選択）を要件に明記すべき
- [x] 1サイクルで完了できる工数か
- [x] EDINET API の制約（無料・要APIキー）を把握しているか
