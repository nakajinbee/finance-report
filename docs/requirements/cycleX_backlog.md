# サイクルX（未着手事項の引き継ぎリスト）

まだどのサイクルにも割り当てられていない「次サイクル以降でやる」と決めた事項を集約する
running backlog。特定のサイクル番号に固定せず、新しいサイクルを始めるときにここから
該当する項目を選び、その`cycleN_requirements.md`に正式に組み込む。組み込んだ項目は
このファイルから削除する（重複管理をしない。このファイルが「まだ着手していない」の
唯一の一覧であり続けるようにする）。

各サイクルの要件定義書の「スコープ外（次サイクル以降）」に新たに積み残しが出た場合は、
都度このファイルに追記すること。

まだ「やる」と確定していない、検討段階のアイデアは[docs/ideas/](../ideas/README.md)に置く
（このファイルとの使い分けは`docs/ideas/README.md`参照）。「全企業データの一括DB投入」
「複数企業の同時比較UI」は、より具体的な検討が[docs/ideas/](../ideas/README.md)で
進んでいるため、重複管理を避けてそちらに一本化した。

## 機能面

### 企業一覧画面（SCR-002）の検索履歴表示
- **内容**：企業一覧画面に、過去に検索・閲覧した企業の履歴を表示する
- **出所**：ユーザー要望（cycle4で発生、cycle4→cycle5で持ち越し済み、未着手）

### テキスト開示（事業の内容等）の保存・検索
- **内容**：現状は数値データ（財務諸表・指標）のみを扱っており、XBRLのテキスト開示部分
  （事業の内容、リスク情報等）は対象外。これを保存・検索できるようにする
- **出所**：cycle2の要件定義でスコープ外と明記（未着手）
- **優先度上昇の可能性**：cycle11で確定したアプリコンセプト
  （[product/concept.md](../product/concept.md)）の提供価値に「定性的な財務状況・
  経営状態の分析」が含まれており、本項目はそれに直結する。サイクル12の
  ユースケース設計（IDEA-13）で具体的な優先度を判断する

### 「タグ不一致で取得できない」と「開示していない」の画面上での区別
- **内容**：現状はどちらも「データなし」表示で統一されている。EDINETのタグ命名が
  会計基準・企業ごとに揺れることで取得できないケースと、その企業がそもそも開示していない
  ケースを、画面上で区別できるようにする（SCR-004の生データ確認画面を見れば判別は可能だが、
  一覧・詳細画面上での即時判別はできない）
- **出所**：cycle3の要件定義でスコープ外と明記（未着手）

### US GAAP企業のさらなる追加検証
- **内容**：cycle3でUS GAAP企業2社（うちオリックス追加）を実機検証済みだが、
  タグの揺れ・カバー範囲の網羅性としては限定的。必要になった時点でさらに検証を広げる
- **出所**：cycle3の要件定義でスコープ外と明記（優先度低、必要が生じたら着手）

## 非機能・パフォーマンス面

### BATCH-004（書類本体取り込み）の全社・全件規模での本実行
- **内容**：サイクル9でBATCH-004は30件サンプルでの動作確認までを実施した。
  `documents`に記録済みの未取得書類（約41,500件規模）全件を対象に本実行する
- **出所**：cycle9の要件定義でスコープ外と明記（数万件規模のEDINETリクエストが
  必要なため、実行前にユーザーへの明示的な許可確認が必須。
  [cycle-workflow](../../.claude/skills/cycle-workflow/SKILL.md)参照）

### 書類一覧・書類本体取り込みの日次自動化
- **内容**：`ingest_document_list_for_date`・`ingest_document_body`は実装済みだが、
  日次での自動実行（cron等のスケジューリング）が未整備。IDEA-01フェーズ4の完成に必要
- **出所**：cycle9の要件定義でスコープ外と明記

### 定性データ（事業概要・リスク等）を元のXBRL/インラインXBRLから抽出し直す
- **内容**：現在の`xbrl_parser.parse_qualitative_facts`は書類取得API`type=5`
  （CSV、XBRLデータをCSV形式に変換したもの）を読んでいるが、元テキストに表
  （例：事業部門別販売実績の推移）が含まれる場合、CSV変換時点で区切り文字が
  失われ数字が連結してしまい読みにくい（サイクル13で実機確認）。`type=1`
  （元のXBRL、提出書類がインラインXBRLの場合は実質HTML）を取得すればテーブル
  構造を保ったまま抽出できる可能性が高いが、ZIP内のHTML／タクソノミファイルを
  解析する実装が新たに必要で、現状の単純なTSVパース（`xbrl_parser.py`）とは
  規模が異なる
- **出所**：サイクル13 UC-1-1の実装中にユーザーが実機確認で発見。対応するかは
  次にこの領域（事業概要・リスク表示の精緻化）に着手するサイクルで判断する
  （[IDEA-15](../ideas/IDEA-15_screen_display_spec_refinement.md)参照）

### 個別ダウンロードがdocuments.list_dateを不正確な値で上書きしうる
- **内容**：サイクル13 FR-60で、個別ダウンロード機能（`routers/edinet.py`）が
  `documents`テーブルを更新するよう修正した際、`list_date`はEDINETの書類一覧APIの
  実際の取得日ではなく`submitDateTime`の日付部分で代用している（`search_report`が
  どの日付で見つけたかを返さないため）。既にBATCH-003（書類一覧バックフィル）で
  正確な`list_date`が記録済みの書類に対し、後から個別ダウンロードを実行すると、
  この近似値で上書きされてしまう可能性がある。`list_date`はバックフィルの進捗管理用
  （参考値）であり実害は小さいが、根本的に直すなら`search_report`が実際に一致した
  日付を返すよう変更し、既存の`list_date`がある場合は上書きしない設計にする必要がある
- **出所**：サイクル13 UC-1-1の実装セルフレビュー（データ整合性確認）で発見。
  実害が小さいため今回は対応せず記録のみ

### 集計済みキャッシュテーブルの追加
- **内容**：財務分析指標等の集計をリクエストの都度計算するのではなく、事前計算した
  キャッシュテーブルを持つ
- **出所**：cycle2の要件定義でスコープ外と明記（パフォーマンス上の必要が生じたら着手。
  現時点で性能問題は報告されていない）

### モバイル対応
- **内容**：現状はPC専用（想定最小幅1280px）。スマートフォン・タブレットでの表示最適化
- **出所**：cycle1・cycle2から一貫してスコープ外。design_guideline.mdのレイアウト方針
  （PC専用）とも整合しており、当面着手の予定なし（優先度は低いが、方針転換の可能性を
  残すため記録は残す）
