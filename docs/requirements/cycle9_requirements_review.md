## 2. 実現可能性チェック

### 技術的実現可能性
- [x] EDINET APIで要件のデータが取得できるか　→ 書類一覧API・書類取得APIとも
  既存のダウンロード機能で実績があり、新規の疎通確認は不要
- [x] EDINET APIの制約　→ NFR-21で試算済み。書類一覧APIは日付単位で必ず1回のみの
  リクエストになるため、レート制限への影響は現行方式より小さい
- [x] XBRLから必要な勘定科目を取り出せるか　→ 既存の`xbrl_parser`をそのまま使うため
  非該当

### サイクル内の工数
- [x] 1サイクルで完了できる量か　→ 新テーブル1つ・処理2つ（一覧取り込み・本体取り込み）
  と、他サイクルより大きめだが、FR-50の全社実行を明示的にスコープ外としたことで
  サイクル内に収まる範囲に抑えている
- [x] 工数が多すぎる場合のスコープ削減判断　→ 上記の通り実施済み

### 依存リスク
- [x] 外部サービスの障害・仕様変更リスク　→ 既存機能と同じリスクレベル
- [x] データが存在しない企業・書類がある場合の対応　→ 上記の通りFR-49に追記済み。
  加えて、`csv_flag='0'`（CSVなし）・取り下げ済み書類はFR-50で除外する設計を
  FR-48・FR-50に明記済み

## 判定：設計フェーズへ進んで問題なし

レビュー中に1件（`sec_code`変換失敗・未知の企業の書類の扱いが未定義だった点）を
発見し、要件定義書に追記済み。修正後、抜け漏れ・矛盾は見つからなかった。

FR-49（書類一覧取り込み、実データで過去10年分を実行）→
FR-50（書類本体取り込み、少数サンプルで動作確認）→ FR-51（ドキュメント整合性確認）。
