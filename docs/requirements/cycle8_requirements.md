# サイクル8 要件定義

## プロダクト概要

サイクル7（[IDEA-01](../ideas/IDEA-01_db_batch_ingestion.md)フェーズ2）の実測で、
現状の実装のまま全上場企業（3,829社）を一括取得すると約8〜10日間の連続稼働が必要と
判明した。同時に、ボトルネックの正体（`fetch_document_list(target_date)`が企業非依存の
結果を返すのに、企業ごとに重複取得している）も特定できている。

本サイクルでは、このボトルネックに対する対策（日付単位キャッシュ）を実装し、
**同条件で再測定して改善効果を検証する**。全社（3,829社）への本番展開そのものは
まだ行わない（改善後の見積もりが実務上許容できる水準か確認してから、次サイクルで
本番展開バッチに着手する）。

## サイクル8の目標

1. `fetch_document_list`の結果を日付単位でキャッシュし、同一プロセス内での重複
   EDINETリクエストをなくす
2. サイクル7と同条件（未取得企業20社、業種分散）で再測定し、改善効果を定量的に確認する
3. 改善後の実測値をもとに全社展開時の所要時間を再試算し、実務上許容できる水準に
   達したか評価する

## 機能要件

### FR-45：`fetch_document_list`の日付単位キャッシュ

- `backend/edinet_client.py`に、`target_date`をキーとするインメモリキャッシュを追加し、
  同一日付への`fetch_document_list`呼び出しが2回目以降はEDINETへリクエストせず
  キャッシュを返すようにする
- キャッシュはプロセスが起動している間保持する（他の既存キャッシュ
  ＝`_filer_info_cache`と同じ方針）。上限・有効期限は設けない
  （探索対象日数は最大でも「探索窓×対象年数×書類種別数」程度＝数千日オーダーで
  収まり、メモリ上問題にならないため。YAGNI）
- `fetch_document_list`の戻り値の型・意味は変更しない（呼び出し側のコード変更は不要）

### FR-46：キャッシュ導入後の再検証

- `backend/scripts/verify_batch_timing.py`のサンプル選定条件に、
  **`accounting_standard IS NULL`（＝未取得企業）であること**を追加する
  （既にサイクル7で取得済みの33社が再度選ばれ、スキップ判定により
  「0秒・0リクエスト」という無意味な結果になることを防ぐため）
- 業種の分散方針（業種ごとに1社、20業種）はサイクル7と同じにする。ただし、
  サイクル7のサンプルと重複しない企業になる（未取得企業に限定するため自然に回避される）
- サイクル7と同じ条件で実行し、所要時間・リクエスト数を計測する

### FR-47：改善効果の評価と見積もりの更新

- FR-46の実測値と、サイクル7の実測値（キャッシュ導入前）を比較し、
  1社あたりの平均所要時間・平均リクエスト数がどれだけ改善したかを算出する
- 改善後の実測値から、全社（3,829社のうち本サイクル終了時点での未取得社数）展開時の
  所要時間を再試算する
- 試算結果が実務上許容できる水準（具体的な閾値は評価時に判断：例えば「1日以内」
  「数時間以内」等）に達したかを評価し、`docs/development/cycle7_batch_timing_estimate.md`
  に改訂版として追記する
- 達していない場合、さらなる高速化案（並列化、`window_days`の縮小等）の要否を
  次サイクルへの申し送り事項として記録する

## 非機能要件

### NFR-18：既存機能への影響

`fetch_document_list`の呼び出し元（`search_report`等）の挙動・戻り値は変わらない
（キャッシュは内部実装の変更のみ）。既存のダウンロード機能（個別企業のダウンロード）が
壊れていないことを実装セルフレビューで確認する。

### NFR-19：レート制限機構への影響

`_wait_for_rate_limit`・`MIN_REQUEST_INTERVAL_SECONDS`は変更しない。キャッシュヒット時は
そもそもEDINETへのHTTPリクエストが発生しないため、レート制限の対象にもならない
（リクエストが減ることでレート制限による待機時間も減る、という間接的な高速化効果はある）。

キャッシュ対象は「本日より前の日付」のみ（`search_report`が既に未来日を候補から除外して
いるため）。過去にEDINETへ提出された書類の履歴は後から変わらないため、キャッシュに
陳腐化のリスクはない。

なお、「書類が見つからない」失敗ケース（サイクル7で1/20社が該当）は、その日付を
最初に探索する企業にとってはキャッシュの恩恵を受けられない（`window_days`の全域を
自分で埋めることになる）。2社目以降が同じ日付範囲を探索する場合はキャッシュが効くため、
サンプル全体・全社展開全体で見れば改善が期待できるが、個々の失敗ケース単体の所要時間が
劇的に短縮されるわけではない点に留意する。
