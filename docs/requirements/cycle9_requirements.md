# サイクル9 要件定義

## プロダクト概要

そもそもの狙い（[IDEA-01](../ideas/IDEA-01_db_batch_ingestion.md)）は、EDINETに都度
問い合わせるとリアルタイム性・応答速度に欠けるため、**自前のDBに先回りしてデータを
登録しておき、画面はDBだけを参照する**というアーキテクチャにすること。

サイクル8で、企業ごとに提出日を推測して検索する現行方式は、EDINETの書類一覧APIの
設計（日付を指定してその日の全提出書類を取得する。企業を指定して取得する機能はない）
に逆行しており、キャッシュで改善しても限界があると判明した。

本サイクルでは、EDINET本来の使い方に沿ったアーキテクチャに作り直す：

1. **書類一覧の取り込み**：毎日（または過去に遡って）EDINETの書類一覧APIを呼び、
   その日に提出された書類のうち対象となるもの（有価証券報告書・半期報告書、
   証券コードを持つ企業）をDBに記録する
2. **書類本体の取り込み**：1で記録した書類のうち、まだ財務データを取得していないものを
   対象に、書類取得APIでCSVを取得・パースし、`facts`に保存する

1と2を分離することで、1（書類の存在を知る）は企業ごとの推測・検索が不要になり、
2（実際のデータ取得）は「どの書類を取ればよいか」が既に分かっている状態から始められる。
また、1の仕組みは「過去に遡って取り込む」（初回一括投入）と「今日の分だけ確認する」
（継続的な更新）の両方に同じコードで対応できる。

## サイクル9の目標

1. 書類一覧APIの結果を保持する新テーブル（`documents`）を設計する
2. 書類一覧を取り込む処理（日付を指定して実行、遡り実行にも継続実行にも使える）を実装する
3. `documents`に記録された書類のうち未取得のものを対象に、実際にCSVを取得し
   `facts`へ保存する処理を実装する
4. 新テーブルの追加が既存のテーブル・API・画面（ユースケース）と整合していることを確認する

## 機能要件

### FR-48：`TBL-004 documents`（書類メタデータテーブル）の新設

書類一覧APIのレスポンスのうち、対象となる書類（`secCode`が存在し、
`docTypeCode`が有価証券報告書(`120`)または半期報告書(`160`)のもの）だけを保存する
（大量報告書・ファンドの書類等、アプリが使わない書類は保存しない。YAGNI）。

主なカラム（詳細は設計フェーズで確定）：

- `doc_id`（主キー、書類管理番号）
- `edinet_code`（提出者のEDINETコード）
- `company_code`（`companies.code`への外部キー、`sec_code`から変換。サイクル6の
  `to_company_code`を再利用）
- `doc_type_code`（書類種別コード）
- `period_start` / `period_end`（対象期間）
- `submit_date_time`（提出日時）
- `list_date`（この書類をどの日付の書類一覧取得で発見したか。書類一覧APIの
  `date`パラメータの値。再取得済みかの判定・進捗管理に使う）
- `withdrawal_status` / `disclosure_status`（取下区分・開示不開示区分。取り下げられた
  書類・非開示の書類を後続処理で除外するために保持する）
- `csv_flag`（CSVの有無。`0`の書類は取得できないため後続処理で除外する）
- `facts_ingested_at`（この書類のCSVを取得し`facts`へ保存済みかどうか。NULL＝未取得）

`company_code`・`facts_ingested_at`は、FR-50（未取得書類の抽出）で頻繁に絞り込み条件に
使うため、インデックスを設ける。

### FR-49：書類一覧の取り込み処理（日付ポーリング）

- 指定した日付（1日、または開始日〜終了日の範囲）について書類一覧APIを呼び、
  条件に合う書類（FR-48参照）を`documents`テーブルにupsertする
- 同じ`doc_id`は重複登録しない（`doc_id`が主キーのため自然に冪等になる）。
  ただし`withdrawal_status`等が変わっている可能性があるため、既存行がある場合は
  値を更新する（登録し直す）
- 過去に遡って実行する場合（初回一括投入）と、直近の日付だけ実行する場合（継続的な
  更新）の両方に同じ関数を使う。範囲実行は日付を1日ずつ進めながらこの処理を呼び出す
- 未来日は指定できないため、上限は「today」までとする
- `sec_code`から`company_code`への変換（`to_company_code`）が末尾`0`以外で失敗する場合、
  その書類はスキップしログに残す（バッチ全体は止めない。FR-39と同じ方針）
- 変換に成功した`company_code`が`companies`テーブルに存在しない場合（`companies`は
  サイクル6時点のEDINETコードリストのスナップショットから作成しており、その後新規上場
  した企業等は含まれていない可能性がある）、その書類もスキップしログに残す。
  `companies`テーブル自体の更新（`bulk_register_companies.py`の再実行）は本サイクルの
  スコープ外とし、`cycleX_backlog.md`に「定期的な`companies`再同期」として積み残す

### FR-50：書類本体の取り込み処理（CSV取得・facts保存）

- `documents`テーブルから、`facts_ingested_at`が`NULL`かつ`csv_flag='1'`かつ
  `withdrawal_status`が取り下げでない行を対象に、書類取得APIでCSVを取得する
- 取得したCSVを既存の`xbrl_parser.parse_numeric_facts`でパースし、`facts`テーブルへ
  保存する（既存の`_upsert_facts`相当のロジックを再利用する）
- 会計基準（`accounting_standard`）が企業に未設定の場合、最初に成功した書類から
  判定して`companies`を更新する（既存の`run_download_job`と同じ考え方）
- 1件の取得・パースに失敗しても処理全体を止めず、次の書類に進む（FR-39と同じ方針）

### FR-51：既存テーブル・画面（ユースケース）との整合性確認

- `docs/design/table/er_diagram.md`に`documents`テーブルとその関係
  （`companies`と1対多、`facts`とは直接の外部キー関係を持たない別系統のテーブル）を追記する
- `docs/design/table/table_list.md`に`TBL-004 documents`を追記する
- 既存の個別企業ダウンロード機能（SCR-001、`run_download_job`）が、`documents`テーブルの
  追加によって壊れないことを確認する。個別ダウンロードは引き続き動作する必要があるため、
  `run_download_job`の内部実装をどう扱うか（そのまま残す／`documents`を経由する形に
  作り替える）は設計フェーズで判断する
- 既存のAPI（`GET /api/companies`等）のレスポンス・挙動に変更がないことを確認する

## 非機能要件

### NFR-20：既存機能への影響

FR-48〜51は新テーブルの追加とバッチ処理の新設が中心であり、既存の画面・APIの
レスポンス内容は変更しない。個別企業のダウンロード機能（SCR-001）は引き続き利用できる。

### NFR-21：EDINETへの負荷

FR-49（書類一覧の取り込み）は1日1リクエストで済むため、過去10年分でも約3,650リクエスト
（試算：約37分、サイクル8の見積もり参照）。FR-50（CSV取得）は対象書類数に比例するが、
「日付を推測して探す」処理が不要になるため、現行方式より大幅に少ないリクエスト数で
済む見込み（1書類＝最短1リクエスト）。

## スコープ外（次サイクル以降、または本サイクル内の実行順序として後回し）

- **全社（3,829社）分のFR-50実行そのもの**：FR-49（書類一覧の取り込み）は本サイクルで
  実際に過去10年分を実行するが、FR-50（CSV取得・facts保存）を全社分実行するのは
  数万件規模の書類取得になり時間がかかるため、本サイクルでは少数サンプルでの動作確認に
  留め、全社分の実行は次サイクル以降（または本サイクル完了後の追加実行）とする
- 定期実行の自動化（cron等での自動起動）：本サイクルはFR-49・50の処理自体を作ることが
  目的で、自動スケジューリングの仕組み化は次サイクル以降とする
- フロントエンドをDB参照のみに切り替える対応（IDEA-01フェーズ5）
- SQLite/MySQL移行判断（IDEA-01フェーズ6）
