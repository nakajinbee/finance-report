# サイクル11 要件定義書

対象：[IDEA-14](../ideas/IDEA-14_business_concept_definition.md)（アプリのビジネス
コンセプト決定）

**前提確認**：本サイクルは通常の「画面・API・バッチの機能追加」とは性質が異なる
企画フェーズ（[cycle-workflow](../../.claude/skills/cycle-workflow/SKILL.md)の
Step 0相当）。成果物はコード・画面の変更ではなく、ユーザーとの対話を通じて確定した
コンセプトのドキュメント化そのものである。そのため本書は通常のFR形式ではなく、
「決定すべき論点」を項目として整理する形式にする。

## プロダクト概要（サイクル10までの状況）

サイクル1〜10でEDINET連携・DB一括化・財務分析指標の実装を進めてきたが、
「誰の・何のためのアプリか」というビジネスコンセプトは未決定のまま機能追加を
続けてきた。IDEA-10（レポート表示の再設計）・IDEA-13（ユースケース設計）は
いずれもこの決定を前提条件としており、着手できない状態が続いていた。

## サイクル11の目標

ユーザーとの対話を通じて、想定利用者・提供価値・開発方針・事業化スタンスを
確定し、`docs/product/concept.md`としてドキュメント化する。以降のサイクル
（サイクル12のユースケース設計、将来のIDEA-10着手時）が参照する一次情報にする。

## 決定した論点

### 論点1：想定利用者

第一想定利用者は開発者自身（会計を学びながら投資判断を行いたい個人投資家）。
「自分が欲しい・便利だと思うものは他人も欲するだろう」という思想のもと、
同様のニーズを持つ個人投資家への展開可能性も見据える（ユーザーインタビュー等は
行わず、開発者自身のニーズを一次情報として機能を決定する＝dogfooding方針）。

### 論点2：提供価値

企業を調べて、定量的（財務諸表・指標）・定性的（事業内容等のテキスト開示）に
財務状況・経営状態を分析し、投資の意思決定につなげる情報収集ができること。
会計を学ぶための実践教材としても機能する。

**発見事項**：「定性的」分析は現状のアプリでは扱っていない領域である。XBRLの
テキスト開示（事業の内容・リスク情報等）は、cycle2の要件定義で意図的にスコープ外
にしたまま`cycleX_backlog.md`に残っている。本コンセプトの確定により、この項目の
優先度が上がる可能性がある（今回のスコープでは着手しないが、backlogに優先度上昇の
根拠を追記する）。

### 論点3：事業化スタンス

現時点では収益化・複数ユーザー対応を目的にしない（自分専用として作る）。ただし
「自分専用でなければ成立しない」実装（単一ユーザー決め打ち等）は避け、後から
ユーザー管理・ログイン機能（IDEA-07）等の入口を追加すれば他者展開ができる
余地は残す、という設計上の配慮を持つ。

## FR-55：コンセプトのドキュメント化

- `docs/product/concept.md`（新規）に、論点1〜3の決定内容を記録する
- `docs/INDEX.md`のディレクトリ構成に`docs/product/`を追加し、
  `design_guideline.md`と同様に「常に最終断面のみを書く」参照ドキュメントとして
  位置づける
- `docs/ideas/IDEA-14_business_concept_definition.md`の状態を「完了」に更新する
- `docs/requirements/cycleX_backlog.md`の「テキスト開示の保存・検索」項目に、
  本コンセプト確定により優先度が上がりうる旨を追記する
