# サイクル15 要件定義 セルフレビュー

[self_review_rule.md](self_review_rule.md)に沿って確認した。

## 抜け漏れチェック

### ユーザーアクション・状態の網羅

- **比較リスト操作**：追加（SCR-003）・削除（SCR-003のトグル、またはSCR-005の
  各列の除外ボタン）の両方をFR-65・FR-66に定義した
- **比較画面の状態**：空（案内メッセージ）・データあり（表表示）の2状態を定義した。
  ローディング・エラーは各企業ごとのAPI呼び出し失敗時にどう見せるかが未定義
  だったため、FR-66に追記が必要 → **修正**：一部の企業のAPI呼び出しが失敗した
  場合、その企業の列にエラー表示をし、他の企業の列は表示を続ける方針とする
  （1社の失敗で画面全体を止めない）
- **ランキング画面の状態**：初期状態（指標未選択、案内メッセージ）・結果あり・
  結果0件（指定した業種に該当企業がない、またはその指標のデータがある企業が
  1社もない）の3状態が必要 → **修正**：FR-68に「結果0件時は『該当するデータが
  ありません』を表示する」を追記する

### 機能の網羅

- 比較リストが`localStorage`のみに保存される設計のため、「シークレットウィンドウ・
  別ブラウザでは比較リストが空になる」という制約がある。これは要件として明記済み
  （FR-65「複数端末での同期は不要」）なので問題ない
- 業種絞り込みの選択肢生成は、SCR-002（サイクル14）で実装済みの
  「取得済み企業一覧から`sector`の重複除去、`null`除外」パターンをSCR-006でも
  再利用する（新規ロジックではなく既存パターンの流用と明記）

### スコープの明確化

- UC-1-3（業界俯瞰）との機能重複リスクを「スコープ外」に明記済み。UC-1-2の
  ランキングはあくまで「UC-1-2の目的（相対位置の把握）」の範囲にとどめ、
  時価総額等UC-1-3固有の観点（IDEA-12の株価データ等）には踏み込まない

## 実現可能性チェック

- ランキングAPI（FR-67）は既存の`_build_financial_records`・`_build_ratio_records`を
  そのまま再利用でき、新規の集計ロジックは不要。技術的な実現性に問題はない
- NFR-15（応答時間）は設計セルフレビュー段階で実測することを明記済み。全社
  約3,800社分のループ処理になるため、実測して問題があれば`cycleX_backlog.md`へ送る
  方針が明確

## 修正内容の反映

上記で見つかった2件（比較画面のAPI部分失敗時の挙動、ランキング結果0件時の
表示）を`cycle15_requirements.md`のFR-66・FR-68に追記した。

## 追記（画面フロー整理段階でのユーザー指摘）

設計フェーズの「画面フローの整理」着手直後、ユーザーから「企業詳細画面
（SCR-003）経由で1社ずつ比較に追加する方式は、比較相手を探しにくい」との
指摘があった。これを受け、比較対象企業の選び方をFR-65として再定義し
（比較企業選択画面＝SCR-005を新設し、SCR-002と同じ検索UXで複数選択してから
比較結果画面へ進む方式に変更）、比較結果画面（FR-66）もグラフ表示要件
（SCR-003と同じカテゴリ構成での企業別棒グラフ）を追加した。要件定義完了後の
早い段階（設計着手直後）で気づけたため、実装前に修正できた。

## 結論

上記の修正を反映済み。設計（画面フロー→画面→API→バッチの順）に進む。
