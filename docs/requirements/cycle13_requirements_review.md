## 1. 抜け漏れチェック

### ユーザーアクション網羅

- [x] UC-1-1-1（財務諸表確認）・UC-1-1-2（財務分析指標確認）は実装済みのため、
  ユーザー操作自体に変更はない（ドキュメント是正のみ）
- [x] UC-1-1-3（定性情報確認）：閲覧操作（開閉可能なセクション）を想定。
  検索・キーワードハイライト等はスコープ外と明記した
- [x] 全状態（成功・失敗・空）：定性情報が存在しない書類（テキストブロックが
  空、または対象書類が半期報告書で一部要素が省略されている場合等）の表示を
  設計フェーズで定義する必要がある（要件定義段階では「開閉可能な形式」とだけ
  規定し、空データ時の表示は設計フェーズで具体化する）

### 機能の網羅

- [x] FR-57（ドキュメント是正）・FR-58（定性情報の取得・保存・表示・遡及適用）で
  UC-1-1の3小項目すべてに対応している
- [x] 機能間の依存：FR-58の遡及適用は既存の`document_body_ingestion`ロジックに
  依存するため、実装順序を設計フェーズで明記する必要がある

### 非機能要件の網羅

- [x] EDINETへの負荷：遡及適用で約515リクエスト・約5分規模になることを明記し、
  ユーザーの事前許可を得た
- [x] データの鮮度：新規に処理される書類は`document_body_ingestion`実行時に
  自動的に定性情報も保存されるため、追加の運用は不要

### スコープの明確化

- [x] UC-1-2・UC-1-3は次段階、定性情報の全文検索機能はスコープ外と明記した

**発見事項**：初稿では既存データへの遡及適用を「要検討事項」として設計フェーズに
先送りする書き方だったが、ユーザーに確認したところ「今回のサイクルでやる」という
判断だったため、要件定義書側を確定事項として修正した（要件定義で確認すべき
論点を後回しにしない、という運用に沿う）。

## 3. FR-59（命名是正）の追加チェック

- [x] 対応表（旧→新）に、テーブル・モデルクラス・関数・ファイル・ドキュメントの
  すべてを網羅しているか確認した（`facts`関連の呼び出し元：`routers/edinet.py`・
  `document_body_ingestion.py`・`scripts/ingest_document_bodies.py`も対象に含めた）
- [x] 既存データ（478,692行）を壊さない設計か：テーブル名の変更のみで行データ・
  カラム構造は変更しないため、データ移行は不要（`ALTER TABLE RENAME`で完結）
- [x] 影響範囲：バックエンドのみ（フロントエンドはAPIレスポンスのフィールド名を
  参照しており、DBのテーブル名・内部変数名を意識しないため、フロントエンドの
  変更は不要）

**発見事項**：当初FR-58で新設予定だった`disclosures`テーブルも、命名是正の
議論の中で「定性データであることが名前から伝わらない」という同種の問題を
抱えていることが分かった。既存の是正（`facts`→`company_quantitative_facts`）と
同時に、新設テーブルも最初から`company_qualitative_facts`という対になる名前で
作ることにし、二重の手戻りを防いだ。
