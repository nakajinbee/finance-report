# 要件定義 セルフレビュールール

要件定義完了後、次フェーズ（設計）に進む前に必ず実施すること。
レビュー結果は `cycle{N}_requirements_review.md` に記録する。

---

## 1. 抜け漏れチェック

### ユーザーアクション網羅
- [ ] ユーザーが行う操作をすべて洗い出せているか
- [ ] 各操作の「入力・処理・出力」が定義されているか
- [ ] 操作の結果として起こりうる全状態（成功・失敗・空）を定義しているか

### 機能の網羅
- [ ] ユーザーの「やりたいこと」に対応する機能がすべて書かれているか
- [ ] 機能間の依存・順序に矛盾はないか
- [ ] エラー・例外時の挙動（データなし・API失敗・タイムアウト）を定義しているか

### 非機能要件の網羅
- [ ] 性能要件（応答時間・同時利用数など）を定義しているか
- [ ] セキュリティ要件（APIキー管理・入力値の扱い）を定義しているか
- [ ] 対応環境（ブラウザ・デバイス）を定義しているか
- [ ] データの鮮度・更新タイミングをユーザーに伝える要件があるか

### スコープの明確化
- [ ] 今サイクルのスコープ内・外を明確に分けているか
- [ ] スコープ外の機能を次サイクル候補として記録しているか

---

## 2. 実現可能性チェック

### 技術的実現可能性
- [ ] EDINET API で要件の指標・データが実際に取得できるか確認したか
- [ ] EDINET API の制約（レート制限・データ形式・カバー範囲）を把握しているか
- [ ] XBRL から必要な勘定科目を取り出せることを確認しているか

### サイクル内の工数
- [ ] 1サイクルで実装・テストまで完了できる量か
- [ ] 工数が多すぎる場合、スコープを削減できるか判断したか

### 依存リスク
- [ ] 外部サービス（EDINET API）の障害・仕様変更リスクを認識しているか
- [ ] データが存在しない企業・書類がある場合の対応を考慮しているか
