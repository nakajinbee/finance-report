## 1. 抜け漏れチェック

### ユーザーアクション網羅
- [x] ユーザーが行う操作をすべて洗い出せているか　→ 本サイクルはデータ取得精度の改善であり、
  新規のユーザー操作は発生しない（検索・ダウンロード・閲覧というサイクル2までの操作のまま）。
  この旨を「画面・API契約への影響」セクションに明記済み
- [x] 各操作の「入力・処理・出力」が定義されているか　→ 上記の通り、入力・操作は変わらない。
  出力は「データなしだった項目に値が入るようになる」「決算日パース失敗でブロックされていた
  企業がダウンロードできるようになる」という改善のみで明記済み
- [x] 操作の結果として起こりうる全状態（成功・失敗・空）を定義しているか　→ FR-17（候補が
  1つも見つからない場合はデータなし）、FR-19（未知の決算日表記は既存動作＝パース不可を踏襲）を
  明記済み

### 機能の網羅
- [x] ユーザーの「やりたいこと」に対応する機能がすべて書かれているか　→ 机上検証で見つかった
  問題点1〜5すべてに対応するFR（17〜20、20は対象外として明記）を用意した
- [x] 機能間の依存・順序に矛盾はないか　→ FR-17・18・19は互いに独立した変更で、依存関係なし
- [x] エラー・例外時の挙動を定義しているか　→ 上記参照。加えてオムニ・プラス・システム・
  リミテッドの404エラーはスコープ外として明記（原因未調査のまま次サイクル送り）

### 非機能要件の網羅
- [x] 性能要件を定義しているか　→ 本サイクルはN/Aと明記（画面変更なし、走査コストは無視できる
  オーダー）
- [x] セキュリティ要件を定義しているか　→ 本サイクルはN/Aと明記（新規外部入力なし）
- [x] 対応環境を定義しているか　→ 本サイクルはN/Aと明記（画面変更なし）
- [x] データの鮮度・更新タイミングをユーザーに伝える要件があるか　→ 本サイクルでは非該当
  （既存のダウンロード時点のデータをそのまま使う、新規のデータ鮮度概念は導入しない）

### スコープの明確化
- [x] スコープ内・外を明確に分けているか　→ FR-20（US GAAP拡充）を対象外として明記
- [x] スコープ外を次サイクル候補として記録しているか　→ 「スコープ外」セクションに3項目
  （US GAAP拡充、タグ不一致と未開示の画面上の区別、404エラー原因調査）を記録済み

## 追記（2026-07-22）：FR-23〜26（財務分析指標の拡充）のセルフレビュー

### 抜け漏れチェック
- [x] ユーザーの「やりたいこと」に対応する機能が書かれているか　→
  「収集可能な指標をできるだけ全部」という要望に対し、実データで確認できた範囲を
  グループA/B/Cに分類して網羅した
- [ ] 各機能の入力・処理・出力が明確か　→ **未確定**。FR-26に明記した通り、
  UI設計（既存グラフに混在させるか別セクションにするか）・API設計（既存エンドポイント拡張か
  新規エンドポイントか）・グループA/Cの自己資本比率の重複時の扱いは設計フェーズで
  決定する必要がある。要件定義の時点では意図的に選択肢を残している
- [x] エラー・例外時の挙動　→ グループB/Cで分母が0またはデータなしの場合の計算不能ケースは
  「データなし」表示とする方針（既存指標と同じ、FR-03踏襲）を明記すべきだったため追記が必要
  → **本レビューで追記した**
- [x] 機能間の依存　→ FR-25はFR-17〜19で確立した「ローカル名候補リスト＋非連結フォールバック」
  の仕組みをそのまま流用する前提であり、依存関係は明確

### 実現可能性チェック
- [x] EDINET APIで指標が実際に取得できるか確認したか　→ 7社の実データで確認済み
  （グループAの一部＝配当性向等は未確認、グループCの棚卸資産回転率は業種により
  欠落率が高い見込みであることも明記済み）
- [ ] 1サイクルで完了できる工数か　→ **要検討**。グループA〜Cで対象指標数が多く、
  UI設計次第では工数が膨らむ可能性がある。設計フェーズで規模を見て、必要なら
  グループ単位で段階的にリリースする判断もありうる

### ドメイン知識との整合性
新たに判明した「同一概念（自己資本比率）が開示値と計算値で二重に存在しうる」という点は
`docs/domain/xbrl_tagging_variability.md`に追記する必要がある（本レビューで追記予定）。

## 判定：設計フェーズへ進んで問題なし（ただし設計判断が複数残っている）

セルフレビューの過程で3点の記載漏れ（NFR項目のN/A明記、FR-19の未知フォーマット時の挙動、
画面・API契約への影響の明記）を発見し、`cycle3_requirements.md`に追記済み。
その他の抜け漏れ・矛盾は見つからなかった。

FR-23〜26については、計算不能時の「データなし」方針を追記した。ただしUI・API設計・
自己資本比率の重複解消については要件定義の時点では未確定であり、設計フェーズで
ユーザーに確認しながら決定する。
