## 1. 抜け漏れチェック

### ユーザーアクション網羅
- [x] ユーザーが行う操作をすべて洗い出せているか　→ 本サイクルは新規のユーザー操作を
  追加しない（バックエンドの一括登録処理と、既存画面での表示調整のみ）
- [x] 各操作の「入力・処理・出力」が定義されているか　→ FR-39（バッチの入力＝CSV、
  出力＝DB登録）、FR-41（画面表示の分岐）を明記済み
- [x] 操作の結果として起こりうる全状態を定義しているか　→ **抜けを発見**：
  企業一覧ページの表示件数が数千件規模に増える影響について記述がなかった。
  「FR-41への補足」として追記し、対策はIDEA-02に委ねる判断を明記した

### 機能の網羅
- [x] ユーザーの「やりたいこと」に対応する機能が書かれているか　→
  企画フェーズで合意した6フェーズのうち、フェーズ1（基本情報の一括登録）に対応する
  FR-39〜41で網羅
- [x] 機能間の依存・順序に矛盾はないか　→ FR-40（スキーマ変更）はFR-39（一括登録）の
  前提であり、実装順もその順で設計フェーズに明記する
- [x] エラー・例外時の挙動を定義しているか　→ **抜けを発見**：`sec_code`重複時の挙動が
  未定義だった。NFR-15として追記し、設計フェーズで実データを見て決める方針にした

### 非機能要件の網羅
- [x] 性能要件　→ NFR-14でEDINETへの追加リクエストが発生しないことを確認。
  フロントの表示性能については「FR-41への補足」で言及
- [x] セキュリティ要件　→ 新規の外部入力・APIを追加しないため対象外
- [x] 対応環境　→ サイクル1〜5から変更なし
- [x] データの鮮度　→ 本サイクルで登録される企業の財務データは「未取得」であることが
  画面上分かるようにする（FR-41）ため対応済み

### スコープの明確化
- [x] スコープ内・外を明確に分けているか　→ フェーズ2〜6・IDEA-03/04を「スコープ外」に
  明記
- [x] スコープ外を次サイクル候補として記録しているか　→
  [ideas/IDEA-01](../ideas/IDEA-01_db_batch_ingestion.md)に既に記録済みのフェーズ構成を
  参照する形にしている

## 2. 実現可能性チェック

### 技術的実現可能性
- [x] EDINET API で要件のデータが実際に取得できるか　→ `EdinetcodeDlInfo.csv`は既存の
  `edinet_client._load_filer_info_cache`で取得・パース済みの実績があり、新規のAPI疎通確認は
  不要（FR-07で実機検証済みの機能を流用するため）
- [x] EDINET API の制約（レート制限等）　→ NFR-14の通り、本サイクルはEDINETへの
  追加リクエストが発生しないため非該当
- [x] XBRLから必要な勘定科目を取り出せるか　→ 本サイクルはXBRL解析を行わないため非該当
  （フェーズ3のスコープ）

### サイクル内の工数
- [x] 1サイクルで完了できる量か　→ バッチスクリプト1本・Alembicマイグレーション1本・
  フロントの表示分岐のみで、他サイクルと比べて小〜中規模
- [x] 工数が多すぎる場合のスコープ削減判断　→ 非該当（工数は妥当な範囲）

### 依存リスク
- [x] 外部サービスの障害・仕様変更リスク　→ `EdinetcodeDlInfo.csv`のフォーマットは
  既存コードが前提とする形式に依存しており、EDINET側の仕様変更リスクは既存機能
  （FR-07企業検索）と同一。新規リスクではない
- [x] データが存在しない企業・書類がある場合の対応　→ FR-41で「財務データ未取得」の
  表示を対応済み。NFR-15で`sec_code`重複という追加のデータ品質リスクを新たに特定した

## 3. 追記レビュー：「この記載だけで設計に落とせるか」の観点での再確認

ユーザー指示により、要件定義書の記載内容だけで設計フェーズにそのまま進めるか（読み手が
実装を推測で補う必要がないか）を、実際のコードと突き合わせて再確認した。

- [x] FR-41「詳細画面で財務データ0件でも問題なく表示できることを確認する」→
  **抜けを発見**：`CompanyDetailPage.tsx`を実機確認したところ、
  `{financials.data.length > 0 && (...)}`により0件時は本当に何も描画されない
  （既存の空状態表示ロジックは存在しない、という当初の想定が誤りだった）。
  「確認する」という検証寄りの表現のままでは設計フェーズが新規UI実装の必要性を
  見落とす恐れがあったため、「新規に実装する」と明記し、現状のコードの問題箇所も
  具体的に記載した
- [x] FR-39「バッチ処理」→ **抜けを発見**：数千行規模のCSVを処理する際、個別行の
  エラーでバッチ全体が止まるかどうかが未定義だった。「1行の不備で全体を止めず、
  スキップしてログに残す」という方針を追記した
- [x] FR-40「関連ドキュメントの更新」→ **抜けを発見**：スキーマ変更の反映先として
  `schemas.py`・`openapi.yaml`のみを挙げており、`docs/design/table/TBL-001_companies.md`
  （カラム定義表で`accounting_standard`が必須のまま）の更新が漏れていた。追記した
- [x] FR-40「既存のダウンロードフロー」→ 変更不要と明記済みで、設計フェーズが
  誤って手を入れることを防げている

## 判定：設計フェーズへ進んで問題なし

レビュー中に計5件（企業一覧の表示件数増加への言及漏れ、`sec_code`重複時の挙動未定義、
財務データ0件時のUI未実装、バッチのエラー処理未定義、TBL-001ドキュメント更新漏れ）を
発見し、要件定義書を修正済み。修正後は、実装者が推測で補う必要のある曖昧な記述は
見当たらず、この記載内容のまま設計フェーズに進める状態になっている。
実装順（設計フェーズで具体化）：FR-40（スキーマ変更）→ FR-39（一括登録バッチ）→
FR-41（フロント表示調整）。
