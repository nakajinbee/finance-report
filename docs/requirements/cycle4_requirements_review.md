## 1. 抜け漏れチェック

### ユーザーアクション網羅
- [x] ユーザーが行う操作をすべて洗い出せているか　→ 本サイクルは新規のユーザー操作を
  追加しない（ヘッダーの「企業一覧」リンク以外は既存操作のまま）。ヘッダーリンクの
  クリック→画面遷移という操作をFR-29に明記済み
- [x] 各操作の「入力・処理・出力」が定義されているか　→ ヘッダーリンクは単純な画面遷移のみ
- [x] 操作の結果として起こりうる全状態を定義しているか　→ DBが空の場合の挙動は既存の
  SCR-002表示に委ねる旨を明記済み

### 機能の網羅
- [x] ユーザーの「やりたいこと」に対応する機能がすべて書かれているか　→
  「ヘッダー・フッターの作成」「グラフを見やすくする」「仮のシンプルなデザイン」の
  3点がFR-29〜32に対応
- [x] 機能間の依存・順序に矛盾はないか　→ FR-32（配色定義）はFR-29〜31が参照する
  前提のため、設計・実装順はFR-32→FR-29〜31の順で進める
- [x] エラー・例外時の挙動を定義しているか　→ 本サイクルは表示・レイアウトの変更のみで、
  新規のエラーケースを発生させない

### 非機能要件の網羅
- [x] 性能要件　→ NFR-09で「バックエンドに変更なし」と明記、フロントの表示速度への
  影響は無視できる程度（対象外と判断）
- [x] セキュリティ要件　→ 新規の外部入力・APIを追加しないため対象外
- [x] 対応環境　→ サイクル1〜3から変更なし
- [x] データの鮮度　→ 本サイクルでは非該当

### スコープの明確化
- [x] スコープ内・外を明確に分けているか　→ 「スコープ外」節に3項目（デザインコンセプト、
  企業一覧の検索履歴表示、全社データ一括投入）を明記
- [x] スコープ外を次サイクル候補として記録しているか　→ 記録済み

## 2. 実現可能性チェック

- [x] 技術的実現可能性　→ Tailwind CSS v4の`@theme`機能、Reactの共通レイアウト
  （`Outlet`パターン）はいずれも既存の技術スタックで対応可能。EDINET APIへの
  影響はないため、外部APIの制約は非該当
- [x] 1サイクルで完了できる工数か　→ ヘッダー・フッター・配色集約・グラフの見た目調整のみで、
  新規のデータ取得ロジックを伴わないため、他サイクルより小規模
- [x] 依存リスク　→ 外部サービス（EDINET）への依存なし

## 追記（2026-07-22）：FR-33のセルフレビュー

- [x] ユーザーの「やりたいこと」に対応する機能が書かれているか　→
  「作り込み感を出したい」という抽象的な要望を、利用シーン・幅・カード化・ボタン・
  空状態表示・配色という具体的な変更点に分解した
- [x] 機能間の依存　→ 配色（`lib/theme.ts`）の変更は`Panel`・`Button`等の新規コンポーネントが
  参照する前提のため、設計・実装順は配色→共通コンポーネント→各画面への適用の順とする
- [x] スコープの明確化　→ 配色は「後から変更してよい」というユーザーの明示的な判断のもと
  今回調整する。本格的なデザインコンセプト決定（サイクル5・6）とは別軸の作業であることを
  要件文に明記済み
- [x] 非機能要件　→ バックエンドへの影響なし（NFR-09と同じ）。既存機能への影響なし
  （NFR-10と同じ、表示・スタイルの変更のみ）

抜け漏れ・矛盾は見つからなかった。

### 追記：レスポンシブ・2列表示の要件レビュー
- [x] どの画面のどの要素を2列にするか具体的か　→ B/S・P/L、財務分析指標4カテゴリ、
  企業一覧カードに対象を限定して明記済み。CFセクションは対にならないため1列のままと
  明記し、あいまいさを残していない
- [x] ブレークポイントの根拠　→ 想定最小幅1280pxに合わせてTailwindの`xl`ブレークポイントを
  採用する根拠を明記済み
- [x] 既存のモバイル非対応方針との整合　→ 「PC専用」の方針は変わらず、2列/1列の切り替えは
  レスポンシブ対応であってモバイル対応ではないことを区別できている

## 判定：設計フェーズへ進んで問題なし

抜け漏れ・矛盾は見つからなかった。実装順（FR-32→FR-29〜31、FR-33は配色→共通コンポーネント→
各画面適用）を設計フェーズで具体化する。
