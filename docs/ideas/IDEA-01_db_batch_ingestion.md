# IDEA-01：データ取得方式の刷新（都度参照→一括DB化）

## ジャンル
エンジニアリング

## 概要
EDINETから都度データを参照する現在の方式をやめ、全企業データを一括取得してDBに登録し、
アプリは常にDBを参照する形にする。

## 詳細
**2026-07-23実機確認**：画面表示（企業一覧・企業詳細等）は現状から一貫してDBのみを参照しており、
EDINETへの通信は発生しない（`routers/companies.py`の全エンドポイントがDBのみ参照している
ことを確認済み）。EDINETに実際に通信するのは`routers/edinet.py`（ダウンロード画面・企業検索）
のみで、これはユーザーが明示的にダウンロード操作をした時だけ。「画面表示のたびにEDINETを
参照している」というのは誤った推測だった（旧記述、下記に打ち消し線で残す）。

フェーズ5の実際の狙いは、「画面がEDINETを見るのをやめる」ことではなく、
**「ユーザーがダウンロード操作をしなくても、最初から全上場企業のデータがDBに揃っている
状態にする」**こと。現状はダウンロード済みの企業だけがDBにデータを持ち、それ以外は
基本情報（サイクル6のフェーズ1）のみで財務データが空という状態のため。

~~現状は画面表示のたびにEDINET APIを参照する構成になっている（詳細は実装時に再確認）。~~
これをやめ、全上場企業のデータをバッチ的に一括取得してDBに保存し、画面表示は常にDB参照のみで
完結させる方式に変更したい。

`cycleX_backlog.md`にあった「全企業データの一括DB投入・SQLiteからの移行」を、より具体的に
検討するもの（重複管理を避けるためbacklog側からは削除しこちらに一本化）。

企業一覧のソート・絞り込み機能（IDEA-02〜04）は、DB化されていた方がSQLクエリで実装しやすく、
このアイデアとの相性が良い。先にDB化してから一覧機能を作る順序が自然。

## メモ
- サイクル7の技術検証で、現状の実装のまま全社（3,829社）を処理すると約8〜10日間の
  連続稼働が必要という結果が出た（実務上許容できない）
- サイクル8で`fetch_document_list`（書類一覧API）の結果を日付単位でキャッシュする
  改善を実装・再検証したところ、約48%の時間短縮（約4.7日）を確認できたが、
  それでも実務上許容できる水準には達しなかった
- **サイクル8で判明した根本原因**：EDINETの書類一覧APIは「日付を指定してその日の
  全提出書類を取得する」設計であり、「企業を指定して取得する」機能はAPI側に存在しない
  （`EDINET_API_仕様書.pdf`で確認済み）。現行の実装（企業ごとに提出日を推測して
  探す）はこの設計に逆行しており、キャッシュで改善しても限界がある
- **フェーズ3は次サイクルで、「日付を毎日ポーリングして書類一覧を取り込む」方式に
  アーキテクチャを変更して要件定義からやり直す**。この方式なら過去10年分
  （約3,650日）を約37分で処理できる試算（現行方式の桁違いに速い）。この方式は
  フェーズ4（定期更新）とも同じ仕組みで実現できる。書類のメタデータ
  （docID・secCode・docTypeCode・period等）を保存する新テーブルの設計が必要
  （詳細：[cycle7_batch_timing_estimate.md](../development/cycle7_batch_timing_estimate.md)の
  「サイクル8：キャッシュ導入後の再検証結果」）
- SQLiteのままで足りるか、他のデータベースへの移行が必要かは、フェーズ6でデータ量の
  実測・見積もり（1年あたりの増加量含む）をもとに検証する
- IDEA-02〜04（企業一覧のUI改善）の前提になりうる
