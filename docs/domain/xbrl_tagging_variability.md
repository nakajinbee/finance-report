# XBRLタグ・開示内容の企業間ばらつきに関するドメイン知識

サイクル3で複数企業（武田薬品工業・トヨタ自動車・良品計画・大本組・正興電機製作所・
太陽化学・フジックス）を実機検証した結果、判明したドメイン知識をまとめる。
`docs/domain/accounting_standards.md`・`会計基準/会計基準の基礎知識.md`は「会計基準が
3種類あり、企業ごとにどれを採用するか」を扱うが、本ドキュメントは
**「同じ会計基準を採用していても、企業によってXBRLタグ・開示内容がどう変わりうるか」**
を扱う（会計基準の種類とは別軸の話）。

要件定義・設計時にこのドキュメントを参照し、新たに実機検証で判明した事象があれば
随時追記すること。

---

## 1. 企業固有拡張タグ（Extension Taxonomy）

### 何が起きるか
EDINETのXBRL提出データは、EDINET共通拡張タクソノミ（`jpcrp_cor:`・`jpigp_cor:`・
`jppfs_cor:`等の名前空間）を基本とするが、**企業が独自に追加定義した拡張タグ**
（企業固有タクソノミ）を使うこともできる。この場合、要素IDの名前空間が
`{報告書コード}_{EDINETコード}:{ローカル名}`という形式になる。

**実機確認例（トヨタ自動車、EDINETコード`E02144`）**：
売上高に相当する値が、共通タグ`jpcrp_cor:RevenueIFRSSummaryOfBusinessResults`ではなく、
`jpcrp030000-asr_E02144-000:OperatingRevenuesIFRSKeyFinancialData`という
トヨタ自社の名前空間を持つ拡張タグで開示されていた（値：48,036,704百万円、実際の売上収益と一致）。

### なぜ起きるか
XBRLの仕組み上、企業は「共通タクソノミにない、自社固有の開示項目」を表現するために
拡張タグを定義できる。大企業・独自の経営指標を重視する企業ほど、経営指標等サマリーの
一部を自社拡張タグで表現する傾向があると推測される（本サイクルでの検証事例は限定的であり、
この傾向自体は仮説の域を出ない）。

### 実装上の考慮点
- 拡張タグの名前空間には提出企業のEDINETコードが含まれるため、**要素IDの完全一致では
  絶対に他社と一致しない**（1社ごとに個別対応が必要になってしまう）
- CSVの「項目名」列は、拡張タグの場合**空欄になることを確認済み**（ラベルによる名寄せ戦略は
  使えない）
- 一方、要素IDの**コロン以降のローカル名**（`OperatingRevenuesIFRSKeyFinancialData`）は、
  共通タクソノミのタグ名と似た命名規則に従っている。ローカル名の候補リストによる照合であれば、
  企業固有拡張タグも一定程度拾える（サイクル3 FR-17で採用する方針）

## 2. 会計基準内での科目名（タグ）の表記ゆれ

### 何が起きるか
同じ会計基準（例：Japan GAAP）を採用していても、企業によって使う勘定科目名・タグが異なる。

**実機確認例**：
- 売上高：多くの企業は`NetSalesSummaryOfBusinessResults`（売上高）を使うが、
  良品計画（小売業）は`OperatingRevenue1SummaryOfBusinessResults`（営業収益）を使っていた
- 純利益：連結子会社を持つ企業は`ProfitLossAttributableToOwnersOfParentSummaryOfBusinessResults`
  （親会社株主に帰属する当期純利益）を使うが、連結子会社を持たない大本組は
  `NetIncomeLossSummaryOfBusinessResults`（当期純利益）を使っていた
  （「親会社株主に帰属する」という概念自体が、連結子会社がなければ発生しないため）

### なぜ起きるか
業種慣行（小売・サービス業は「営業収益」、製造業は「売上高」を好む傾向があるとされる）や、
連結・非連結という会社の構造上の違いによる。

### 実装上の考慮点
- 表記ゆれは「同一の会計概念を指す別名」であることが多く、候補リストへの追加で対応してよい
- ただし「経常利益」と「営業利益」のように、**会計上明確に異なる概念**を安易に代替してはならない
  （本サイクルの机上検証では、大本組の営業利益は主財務諸表セクションに`jppfs_cor:OperatingIncome`
  として別途存在していたため、経常利益での代替は不要だったが、これは大本組固有の結果であり
  一般化はできない。将来、真に営業利益を開示しない企業が見つかった場合は、代替可否を
  都度judgement要）

## 3. 連結子会社を持たない企業のコンテキストID

### 何が起きるか
経営指標等サマリー等の数値には、コンテキストID（期間・連結/非連結の区分）が付与されている。
連結子会社を持たない企業は、連結ベースのコンテキストID（`CurrentYearDuration`・
`CurrentYearInstant`）ではなく、**非連結（個別）を示すサフィックスが付いたコンテキストID**
（`CurrentYearDuration_NonConsolidatedMember`・`CurrentYearInstant_NonConsolidatedMember`）
で数値が提出される。

**実機確認例（大本組、建設業、EDINETコード`E00255`）**：
要素ID自体は共通タグ（`jpcrp_cor:NetSalesSummaryOfBusinessResults`等）のままだったが、
コンテキストIDが`_NonConsolidatedMember`付きになっており、非連結考慮なしのマッピングでは
全指標が取得漏れになっていた。

### なぜ起きるか
連結財務諸表は「連結子会社が存在する場合」に作成が求められる。連結子会社を持たない
（あるいは規模が小さく連結の必要がない）企業は、個別財務諸表のみを提出するため、
経営指標等サマリーも個別ベースの数値のみになる。

### 実装上の考慮点
- 要素IDは共通タグのままであることが多いため、**コンテキストIDのみのフォールバック**
  （非連結サフィックスの有無を両方試す）で対応できる（サイクル3 FR-18で採用する方針）
- 連結・非連結どちらのコンテキストで見つかったかは、**コンテキストIDの
  `_NonConsolidatedMember`サフィックスの有無**で判別する。TBL-003
  `facts.consolidated_or_individual`列（CSVの「連結・個別」列をそのまま保存したもの）は
  この判別には使えない。大本組の実データで確認したところ、`_NonConsolidatedMember`
  コンテキストの行でも`consolidated_or_individual`の値が`"個別"`・`"その他"`など
  一定しておらず、信頼できる判別材料ではないことが判明した

## 4. 決算日表記のゆれ

### 何が起きるか
EDINETコードリストの「決算日」列は、通常`N月N日`形式（例："3月31日"）だが、
日を明記せず**「N月末日」**（例："8月末日"）と表記される企業がある。

**実機確認例**：良品計画（EDINETコード`E03248`）の決算日は`8月末日`。

### なぜ起きるか
定款上、決算日を「毎年N月の末日」と定めている企業がある（暦年のうるう年等によって
実際の日付が変わりうるため、日付を固定せず「月末」とすることで、閏年の2月末等にも
柔軟に対応できるようにしていると推測される）。

### 実装上の考慮点
- 「月末」が実際に何日になるかは対象年によって変わる（うるう年の2月等）ため、
  パース時点で日付を確定させることができない。「月末」という情報を保持し、
  対象年が確定した時点（ダウンロード対象年度の決定時等）で実際の日付に解決する必要がある
  （サイクル3 FR-19で採用する方針）
