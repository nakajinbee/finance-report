## データ量の見積もり（フェーズ6：SQLite/MySQL判断向け、追加データ取得なしで実測値から算出）

本サイクルの検証実行で得られた実データ（33社分の`facts`）をそのまま使い、追加の
EDINETアクセスなしで算出した。あくまで概算だが、フェーズ6（DB移行判断）・フェーズ3
（取り込みバッチの設計）双方に使える一次情報として残す。

| 項目 | 値 |
|---|---|
| 現在のDBファイルサイズ | 128.8 MB（企業マスタ3,829社＋facts 33社分） |
| facts取得済み企業数 | 33社 |
| facts総行数 | 310,483行 |
| 1社あたり平均facts行数 | 約9,409行 |
| factsの1行あたりの概算バイト数 | 約435バイト（DBファイルサイズ÷facts総行数） |
| **全社（3,829社）換算の推定facts行数** | **約3,602万行** |
| **全社換算の推定DBサイズ** | **約14.6 GB** |

### 評価（MySQL移行判断に向けて）

約3,600万行・約15GBという規模は、SQLiteが技術的に扱えない規模ではない
（SQLiteは仕様上テラバイト級・数十億行まで扱える）。したがって、**「サイズが理由で
SQLiteでは足りない」とは言えない**。MySQL等への移行を検討する動機があるとすれば、
サイズそのものよりも次のような運用面の理由になる：

- フェーズ4（定期更新バッチ）稼働時に、書き込み中の同時読み取り（アプリからの
  参照）が発生する。SQLiteはファイルロックベースで同時書き込みに弱く、
  複数プロセスからの並行アクセス（フェーズ3で検討中の並列化とも関係する）を
  行うならMySQL等の方が適する
- バックアップ・レプリケーション等の運用のしやすさ

**結論**：データ量そのものはSQLite継続の障害にならない。移行するかどうかは
「同時書き込み・並列処理をどこまでやるか」という運用設計（フェーズ3・4の設計内容）
次第であり、フェーズ6は「データ量」ではなく「並行アクセスパターン」を主な判断材料に
すべき、という見立てに更新する。

## 評価

この所要時間は実務上許容できない（サーバーを1週間以上専有し続けることになり、
途中でEDINET側の障害・アプリの再起動等が起きれば最初からやり直しのリスクも大きい）。
フェーズ3をこのまま「フェーズ1と同じ発想の一括バッチ」として実装するのは現実的でない。

さらに、失敗ケース（20社中1社、5%）は`search_report`が`window_days`の範囲を
最後まで探索してから諦めるため、成功ケースより顕著にコストが高い
（491.9秒/610リクエストで最大値）。全社規模では約190社（3,829社の5%）がこの
最悪パターンに該当する可能性があり、上記の試算より実際にはさらに時間がかかる
可能性が高い。

## フェーズ3への申し送り事項

### 発見した最大の改善余地：書類一覧APIの結果は企業非依存でキャッシュできる

`edinet_client.fetch_document_list(target_date)`は**指定日にEDINETへ提出された
全書類の一覧**を返す。company_codeやedinet_codeに依存しない。現在の実装は
`search_report`が企業ごとに独立して同じ日付の一覧を何度も取得しており、
決算期が近い企業同士（例：3月決算企業が多数）では、同じ日付の書類一覧を
何十回・何百回も重複して取得している可能性が高い。

**フェーズ3では、`fetch_document_list`の結果を日付単位でプロセス内キャッシュし、
全社のバッチ処理を通じて使い回す設計にすることを強く推奨する**。これだけで
リクエスト数を大幅に削減できる可能性がある（削減率はキャッシュ実装後に実測が必要）。

### その他の検討事項

- **並列化**：現在のレート制限（`MIN_REQUEST_INTERVAL_SECONDS`）はプロセス内のグローバル
  変数で管理されており、複数プロセスで並列実行した場合に実際のEDINET側のレート制限を
  超えないかは未検証。上記のキャッシュ導入後に、なお時間が足りない場合の次善策として検討する
- **`window_days`の縮小**：デフォルト25日が本当に必要か、実際の提出日のズレ幅を
  今回のログから分析し、縮小の余地がないか確認する（本サイクルでは個々の探索日数までは
  ログに残していないため、次回検証時に追加計測する）
- **失敗ケースの原因調査**：オムニ・プラス・システム・リミテッド（`code=7699`）は
  FR-21（サイクル3）で一度対応した既知の404企業。今回のサンプルにたまたま含まれており、
  「見つからない」ケースの実測データとして有用だった
- **段階的ロールアウト**：一度に全社を処理せず、優先度の高い企業群から段階的に
  処理するアプローチも検討に値する（例：業種・時価総額等での優先順位付け）

キャッシュ導入によるリクエスト削減効果を測ることが、フェーズ3着手前の
次の技術検証として最も価値が高い。
