# サイクル10：データベース移行判断（IDEA-01フェーズ6）

対象：[cycle10_design.md](../design/cycle10_design.md) FR-52〜54
関連：[IDEA-01](../ideas/IDEA-01_db_batch_ingestion.md)フェーズ6

---

## 結論：SQLiteを継続する

データ量・並行アクセスパターンのいずれの観点からも、現時点でMySQL等への移行が
必要になる理由はない。判断が覆る条件は本文末尾に記載する。

---

## 1. データ量の実測・全社換算推定（FR-52）

### 実測値（2026-07-23時点、`financials.db`）

| 項目 | 値 |
|---|---|
| `facts`テーブル＋関連インデックスの実サイズ | 208,904,192 bytes（約199.2 MB） |
| `facts`総行数 | 482,692行 |
| `facts`に含まれる書類数（distinct doc_id） | 515件 |
| 1書類あたり平均facts行数・サイズ（インデックス込み） | 約937行・約405,639 bytes（約396 KB） |
| `documents`テーブル＋関連インデックスの実サイズ | 7,176,192 bytes（約6.84 MB） |
| `documents`総行数（過去10年分バックフィル済み、全社分完了） | 41,567行 |
| 全期間ダウンロード済み企業（docs≧2、52社）の1社あたり平均書類数 | 約9.3件（min 2・max 12） |
| SQLite `journal_mode` | `delete`（デフォルトのロールバックジャーナル。WALではない） |
| SQLite `busy_timeout` | 5000ms |

現状の`facts`保持企業83社の内訳は、サイクル7/8の技術検証で複数期間を取得済みの
52社（1〜12書類/社）と、サイクル9のFR-50サンプルで1書類のみ取得した31社が
混在している。そのため「facts行数÷社数」の単純平均は取得済み書類数の偏りで
歪むと判断し、「facts行数÷書類数（＝約937行/件）」を基準単位にして以下を算出した。

### 全社換算の推定（初回バックフィル）

- 推定書類数：3,829社 × 約9.3件/社 ≈ **約35,600件**
- 推定`facts`サイズ：35,600件 × 約396KB/件 ≈ **約14.1 GB**

サイクル7時点の見積もり（33社サンプル、facts行数ベースで約14.6GB）とほぼ一致した。
サンプルが33社→83社（かつ書類数ベースの単位に修正）に変わっても同水準の見積もりに
収束したことで、この規模感の信頼性は高いと判断する。

`documents`テーブルは既に全社分の10年分バックフィルが完了済み（41,567行・
約6.84MB）であり、初回分としての追加増加はない。

### 定常運用時の年間増加量

初回バックフィル完了後、各社は年1件（有価証券報告書）〜2件（＋半期報告書）の
新規書類を提出する。

- `facts`：3,829社 × 1〜2件/年 × 約396KB/件 ≈ **年間約1.5GB〜3.0GB**
- `documents`：3,829社 × 1〜2件/年 × 約173bytes/件（`documents`実測：
  7,176,192 / 41,567 ≈ 173bytes/行） ≈ 年間1MB未満（無視できる規模）

### 評価

約14GB規模（初回バックフィル完了時点）・年間数GB程度の増加は、SQLiteが技術的に
扱えない規模ではない（SQLiteは仕様上テラバイト級・数十億行まで扱える）。
**データ量そのものは移行の理由にならない。**

## 2. 並行アクセスパターンの整理（FR-53）

サイクル9で実装した書き込み処理（`document_list_ingestion`・
`document_body_ingestion`）を前提に整理する。

| 処理 | 書き込み頻度・粒度 |
|---|---|
| 書類一覧の取り込み（`ingest_document_list_for_date`） | 1日1回、1トランザクションでまとめてcommit。将来の日次自動化後もEDINETの1日あたり対象書類数は数十件程度で、短時間の書き込みになる見込み |
| 書類本体の取り込み（`ingest_document_body`） | 1書類ごとに1コミット。意図的に書類単位でコミットしており、1回のロック保持時間は短い |
| 個別ダウンロード（`routers/edinet.py`） | ユーザー操作時のみ、期間ごとに1コミット。頻度は低い |
| FastAPIの読み取り（`routers/companies.py`等） | リクエストの都度、読み取りのみ |

`journal_mode='delete'`はコミット時に短時間の排他ロックを取る方式で、WALモードほど
「書き込み中の読み取り」に強くはない。ただし`busy_timeout=5000`が既に設定されており、
ロック競合が発生してもエラーにはならず、最大5秒間リトライ待機してからアクセスする。

### 評価

- 書き込みは「1日1回の短時間バッチ」＋「書類単位でコミットする本体取り込み
  （さらに短時間）」＋「ユーザー操作時のみの個別ダウンロード」であり、いずれも
  長時間のトランザクションを保持しない設計に既になっている
- 読み取り（画面表示API）が書き込みと競合しても、`busy_timeout`により待機後に
  リトライされるため、エラーにはならない（応答が数秒遅延する可能性はあるが、
  致命的な失敗ではない）
- SQLiteが不向きになるのは、複数のアプリケーションサーバー（別マシン・別プロセス群）
  から同時にDBファイルへ書き込む「水平スケーリング」構成になった場合である
  （SQLiteは単一ファイルへの組み込み型DBのため、複数マシンからの同時書き込みという
  構成自体がそもそも成立しない）。現状もサイクル9の設計も単一プロセス（1台のサーバー）
  での運用を前提にしており、複数サーバーでの水平スケーリング計画は存在しない

**現状およびサイクル9で実装した書き込みパターンの範囲では、SQLiteの同時書き込み
制約が実運用上の問題になる可能性は低い。**

## 3. 判断が覆る条件（将来の再検討トリガー）

以下のいずれかに該当する状況になった場合は、この判断を再検討する：

- 複数のアプリケーションサーバー（水平スケーリング）での運用が具体的に計画された場合
- 実測でロック競合による書き込み遅延・タイムアウトが実際に問題として報告された場合
- データ量が本見積もり（約14GB＋年間数GB）を大きく超える運用（対象書類種別の
  拡大等）に変わった場合

---

## 判断

**SQLiteを継続する。** 理由：(1) データ量は約14GB規模で技術的な上限に対し
十分小さい、(2) 現状・サイクル9実装後の書き込みパターンはいずれも短時間・
低頻度で`busy_timeout`により実運用上の問題になりにくい、(3) 水平スケーリング
（複数サーバーからの同時書き込み）の計画が現時点で存在しない。以上3点から
移行の必要性はない。
