# 開発 セルフレビュールール

開発完了後、テストフェーズに進む前に必ず実施すること。
レビュー結果は `cycle{N}_development_review.md` に記録する。

---

## 1. 設計の全実装チェック

- [ ] 設計書に記載されたすべての画面・コンポーネントを実装しているか
- [ ] 設計書に記載されたすべての API エンドポイントを実装しているか
- [ ] 設計書のデータフロー通りに実装されているか
- [ ] カスタマイズ設定（追加・削除・ローカルストレージ保存・リセット）をすべて実装しているか
- [ ] 設計書に記載されたエラー処理フローをすべて実装しているか

---

## 2. 動作確認（実装後に手動で確認）

### 正常系
- [ ] 企業名で検索して結果が表示されるか
- [ ] 証券コードで検索して結果が表示されるか
- [ ] 企業を選択して財務指標が一覧表示されるか
- [ ] 棒グラフが正しく表示されるか
- [ ] 指標の追加・削除ができるか
- [ ] ページを再読み込みしてもカスタマイズ設定が維持されるか
- [ ] デフォルトにリセットできるか

### 異常系
- [ ] 検索で0件の場合にメッセージが表示されるか
- [ ] EDINET API がエラーを返した場合にエラーメッセージが表示されるか
- [ ] 対象指標の XBRL データが存在しない場合に「データなし」と表示されるか
- [ ] ネットワーク切断時に適切なエラーが表示されるか

---

## 3. コード品質

- [ ] APIキー・秘密情報がソースコードにハードコードされていないか
- [ ] ユーザー入力をサニタイズしているか（XSS対策）
- [ ] ブラウザのコンソールにエラー・警告が出ていないか
- [ ] 不要なデバッグログ（console.log など）を削除しているか
- [ ] 型の定義を使っているか（Pythonは型ヒント必須、TypeScriptは`any`を避け明示的な型を使う）
- [ ] メソッド・クラス・変数の名前が、そのユースケースや機能を見て分かる名前になっているか（`data`・`temp`・`process`のような汎用名を避ける）
- [ ] 日付・時刻が[date_format_policy.md](date_format_policy.md)どおりに扱われているか（`date`/`datetime`の混在なし、UTC変換なし、`fiscal_year`を動的生成していないか）
