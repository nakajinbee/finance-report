# セルフレビューガイドライン

各フェーズの成果物を次のフェーズに進む前にセルフレビューすること。
レビュー結果は各フォルダに `*_review.md` として残す。

---

## 要件定義 セルフレビューチェックリスト

### ユーザー視点
- [ ] ユーザーが「やりたいこと」をすべて網羅できているか
- [ ] 検索・表示・操作のすべてのユーザーアクションを列挙できているか
- [ ] エラー時・例外時のユーザー体験を定義しているか（データなし・API失敗など）

### 機能の完全性
- [ ] 各機能に「入力・処理・出力」が明確に書かれているか
- [ ] スコープ外を明記しているか（サイクル2以降に何を回すか）
- [ ] 機能間の依存関係・順序に矛盾はないか

### 非機能要件
- [ ] パフォーマンス要件（応答時間・件数上限）を定義しているか
- [ ] セキュリティ要件（APIキー管理・個人情報など）を定義しているか
- [ ] 対応ブラウザ・デバイスを定義しているか
- [ ] エラーハンドリングのポリシーを定義しているか

### データ
- [ ] 使用するデータソースの制約・限界を把握しているか
- [ ] データが存在しない場合の挙動を定義しているか
- [ ] データの鮮度（いつ時点のデータか）をユーザーに示す要件があるか

---

## 設計 セルフレビューチェックリスト

### アーキテクチャ
- [ ] 要件をすべて満たせる構成になっているか
- [ ] 技術スタックの選定理由が明確か
- [ ] ボトルネックになりうる箇所を特定しているか

### API・データフロー
- [ ] EDINET API のレート制限・制約を設計に反映しているか
- [ ] APIキーの安全な管理方法を決めているか
- [ ] データのキャッシュ戦略を決めているか

### 画面設計
- [ ] 全画面・全状態（ローディング・エラー・データなし）を設計しているか
- [ ] ユーザーフローが要件のユーザーアクションをすべてカバーしているか

### 拡張性
- [ ] サイクル2の追加要件を想定した作りになっているか（壊さずに拡張できるか）

---

## 開発 セルフレビューチェックリスト

### コード品質
- [ ] 要件定義の全機能を実装しているか（実装漏れがないか）
- [ ] エラーハンドリングを実装しているか（API失敗・データなし・ネットワークエラー）
- [ ] ハードコードされた値（APIキー・URLなど）がないか

### 動作確認
- [ ] 正常系（検索ヒット・データあり）が動くか
- [ ] 異常系（検索0件・APIエラー・データ欠損）が動くか
- [ ] ブラウザのコンソールにエラーが出ていないか

### セキュリティ
- [ ] APIキーがフロントエンドのソースに露出していないか
- [ ] ユーザー入力をサニタイズしているか
