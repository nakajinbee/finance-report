# サイクルの回し方

このプロジェクト（企業会計情報WEBアプリ）では、機能追加・改修は必ず以下の順で進める。
**途中のフェーズを飛ばして実装から始めない。** ユーザーから明示的に「実装して」と言われても、
要件定義・設計がまだであれば先にそれを提案する。

## 0. 企画フェーズ（毎回ではなく、方向性が複数ありうる時だけ）

ユーザーの要望が抽象的（「デザインを良くして」「便利にして」等）で、進め方に複数の妥当な
選択肢がある場合は、要件定義に入る前に企画フェーズを挟む。
- 方向性の選択肢を2〜4個、具体的な違いが分かる形で提示する（`AskUserQuestion`を使い、
  可能ならプレビュー付きで）
- ユーザーが選んだ方向性を前提に、要件定義に進む
- 選択肢が1つしかない、または要望自体が既に具体的な場合はこのフェーズを省略してよい

## 1. 要件定義

- `docs/requirements/cycleN_requirements.md`を新規作成
- 構成：プロダクト概要（前サイクルまでの状況、今回の位置づけ）→サイクルNの目標→
  機能要件（FR-xx、連番はプロジェクト全体で通し番号）→非機能要件（NFR-xx）→
  スコープ外（次サイクル以降）
- **`docs/requirements/cycleX_backlog.md`を必ず確認する**。過去のサイクルからの持ち越し
  事項の中に今回拾えるものがあれば、この要件に組み込む（[todo-tracker](../todo-tracker/SKILL.md)参照）
- UI・デザインに関わる要件は、`docs/design/design_guideline.md`の内容と矛盾しないか
  確認する（矛盾する新しい要望があれば、先にガイドライン側の更新が必要かユーザーに確認する。
  [design-apply](../design-apply/SKILL.md)参照）
- 複雑なドメイン知識（会計・EDINETの仕様等）が絡む場合は、要件定義を書く前に
  `docs/domain/`に調査結果を書く

## 2. 要件定義セルフレビュー

- `docs/requirements/cycleN_requirements_review.md`を作成
- `docs/requirements/self_review_rule.md`のチェックリスト（抜け漏れ・実現可能性）を使う
- 見つかった問題は要件定義書自体を修正してから次に進む（レビュー結果に「後で直す」を残さない）

## 3. 設計

- `docs/design/cycleN_design.md`を作成
- FRごとに、対象ファイル・変更前後のコード（diff形式推奨）を具体的に書く。「〜のように直す」
  という曖昧な記述で終わらせず、実装者（未来の自分や別セッションのAI）がそのまま手を動かせる
  粒度にする
- 設計中に既存コードを`grep`等で確認し、想定と違う箇所（未使用コード、対象外にすべき要素等）
  があれば設計書にその判断根拠ごと書く
- 実装順（どのFRから着手するか、依存関係）を明記する

## 4. 設計セルフレビュー

- `docs/design/cycleN_design_review.md`を作成
- `docs/design/self_review_rule.md`のチェックリスト（機能面・性能面・セキュリティ・拡張性）
- **続けて[req-design-traceability](../req-design-traceability/SKILL.md)を実行する**。
  FR単位の対応確認だけでは、FR内の箇条書き1つ1つの見落としや、要件文と設計判断の
  食い違いを見逃す
- 見つかった問題は要件定義書・設計書の両方を実態に合わせて修正してから次に進む

## 5. 実装

- 設計書のdiffどおりに実装する。設計にない変更（ついでの改善等）はしない
- 実装中に設計と実際のコードが食い違う・設計の想定が誤っていたと分かった場合は、
  黙って実装を優先せず、設計書を修正してから実装を続ける
- バックエンドロジックが絡む変更は、モックではなく実際のEDINETデータで動作確認する

## 6. 実装セルフレビュー

- `docs/development/cycleN_development_review.md`を作成（新規サイクルなら新規ファイル、
  既存サイクルへの追記なら該当ファイルに追記）
- `git diff`を設計書のdiffと1件ずつ突き合わせる
- 動作確認：`npx tsc -b --noEmit`・`npx oxlint`が通ること、vite HMRログにエラーが
  出ていないこと、バックエンド・フロントエンドが起動して応答すること（`curl`で確認）
- 実装中にユーザーの実機確認等で設計スコープ外の不具合が見つかった場合は、その場で
  原因調査・修正し、レビュー結果に「設計スコープ外で見つかった不具合」として追記する
- 最後に、今回スコープ外にした事項があれば`cycleX_backlog.md`に追記する
  （[todo-tracker](../todo-tracker/SKILL.md)参照）

## 7. コミット

- **ユーザーから明示的に指示されるまでコミットしない**
- コミット前に`git status`で意図しない変更が含まれていないか確認する
- コミットメッセージは「何を」「なぜ」を簡潔に。サイクル番号・FR番号を含める
