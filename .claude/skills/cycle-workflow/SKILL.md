# サイクルの回し方

このプロジェクト（企業会計情報WEBアプリ）では、機能追加・改修は必ず以下の順で進める。
**途中のフェーズを飛ばして実装から始めない。** ユーザーから明示的に「実装して」と言われても、
要件定義・設計がまだであれば先にそれを提案する。

## 0. 企画フェーズ（毎回ではなく、方向性が複数ありうる時・次サイクルのネタ出しをする時）

新しいサイクルを始める前、または方向性が複数ありうる要望を受けたときは、要件定義に入る前に
企画フェーズを挟む。

- **次に何をやるか自体が決まっていない場合**：まず`docs/ideas/README.md`の一覧を確認する。
  優先度・いつ反映するか（ブロッカーの有無）を踏まえて、今回のサイクルで拾えそうな
  アイデアをユーザーに提示し、相談する。採用が決まったアイデアは、
  [todo-tracker](../todo-tracker/SKILL.md)の手順で`docs/ideas/`から
  `docs/requirements/cycleX_backlog.md`または直接`cycleN_requirements.md`に格上げする
- ユーザーの要望が抽象的（「デザインを良くして」「便利にして」等）で、進め方に複数の妥当な
  選択肢がある場合は、方向性の選択肢を2〜4個、具体的な違いが分かる形で提示する
  （`AskUserQuestion`を使い、可能ならプレビュー付きで）
- ユーザーが選んだ方向性を前提に、要件定義に進む
- やることが既に具体的に決まっている場合はこのフェーズを省略してよい

### ユースケース単位でのサイクル運用（サイクル12以降）

サイクル12（[IDEA-13](../../docs/ideas/IDEA-13_use_case_design.md)）以降、機能追加は
「ユースケース単位」でサイクルを回す。`docs/product/use_cases.md`に大項目・中項目・
小項目（最大3階層）で整理されたユースケース一覧（軽量なインデックスのみ）があり、
1つの中項目（または規模に応じて小項目単位）を1サイクルの対象にする。ユースケースを
まだ実現できていない画面・API・バッチだけを設計・実装し、ユースケースの実現に
不要な既存画面が判明した場合は削除・統廃合も検討する（機能を積み増すだけでなく、
目的に合わない機能を削る判断もこのプロセスに含む）。

各中項目の詳細（画面遷移・関連機能・依存関係等）は`use_cases.md`本体には書かず、
`docs/product/use_case_details/UC-<大項目番号>_<大項目名>/UC-<大項目番号>-<中項目番号>_<中項目名>.md`
に書く（大項目ごとにフォルダを分け、中項目ごとに1ファイル）。詳細ファイルには
「どのサイクルで対応するか」といったスケジュール情報は書かない（`use_cases.md`
同様、常に最終断面のみを書く）。書いてよいのはTODO（後で解消され削除される
一時的な記述）のみ。詳細ファイルが作成済みの中項目のみ、`use_cases.md`側から
リンクする。

ユースケース単位のサイクルでは、設計フェーズ（Step 3）を以下の順で整理する：

1. 画面フローの整理（このユースケースを実現するために、どの画面をどう遷移するか）。
   対象UC-x-xの詳細ファイル（上記）に「画面遷移」節として追記する（新規画面が
   必要か、既存画面への追加で足りるか、遷移が増えるか等）。全画面を俯瞰する
   [flow_diagram.md](../../docs/design/screen/flow_diagram.md)への反映は、
   個々のユースケースでこの整理を終えてから行う（先に図を作らない。ユースケース
   単位の整理が置き去りになる）
2. 画面の整理（各画面の構成・表示内容）。このタイミングで、対象UC-x-xの詳細
   ファイルに「関連機能」として紐づく画面・API・バッチのIDを書き戻す
   （ユースケース確定時点ではまだ紐づけない。画面の整理をするサイクルで
   まとめて追記する）
3. APIの整理（画面が必要とするデータをどのAPIで提供するか）
4. バッチの整理（APIが参照するデータをどう用意するか。既存バッチの再利用も検討する）

この順序で整理することで、「何のためにある画面か」が先に決まり、画面の表示内容・
情報設計の判断基準が明確になる（ユースケース起点ではなく機能起点で作った結果、
目的のない数値表示画面ができてしまったサイクル1〜10の反省を踏まえた運用変更）。

### 企画内容のセルフレビュー（前提確認）

サイクルの内容（何をやるか）を決めるとき、`docs/ideas/`や`cycleX_backlog.md`に書かれた
「現状は〜になっている」「〜という前提で」といった記述を**そのまま鵜呑みにしない**。
アイデアが最初に書かれた時点の推測や、何サイクルも前の状況のまま更新されずに残っている
ことがある（実例：IDEA-01に「画面表示のたびにEDINETを参照している」という未検証の
記述が3サイクルにわたって残り続け、実際には既にDBのみを参照する設計になっていた）。

要件定義を書き始める前に、その前提が今の設計・実装と一致しているかを`grep`・コード読解で
確認する。具体的には：
- アイデア・backlog項目の記述に「現状は〜」という現状認識が含まれる場合、必ず該当する
  コード（バックエンドのルーター・フロントのAPI呼び出し等）を実際に読んで裏取りする
- 前提が古い・誤っていた場合は、要件定義を書く前に元のアイデア・backlog側の記述を
  訂正する（誤った前提のまま要件定義に進まない）
- この前提確認は`docs/requirements/cycleN_requirements_review.md`の冒頭、または
  企画メモに一言残す（「前提確認：〜を実装で確認済み」）

## 1. 要件定義

- `docs/requirements/cycleN_requirements.md`を新規作成
- 構成：プロダクト概要（前サイクルまでの状況、今回の位置づけ）→サイクルNの目標→
  機能要件（FR-xx、連番はプロジェクト全体で通し番号）→非機能要件（NFR-xx）→
  スコープ外（次サイクル以降）
- **`docs/requirements/cycleX_backlog.md`を必ず確認する**。過去のサイクルからの持ち越し
  事項の中に今回拾えるものがあれば、この要件に組み込む（[todo-tracker](../todo-tracker/SKILL.md)参照）
- UI・デザインに関わる要件は、`docs/design/design_guideline.md`の内容と矛盾しないか
  確認する（矛盾する新しい要望があれば、先にガイドライン側の更新が必要かユーザーに確認する。
  [design-apply](../design-apply/SKILL.md)参照）
- 複雑なドメイン知識（会計・EDINETの仕様等）が絡む場合は、要件定義を書く前に
  `docs/domain/`に調査結果を書く

## 2. 要件定義セルフレビュー

- `docs/requirements/cycleN_requirements_review.md`を作成
- `docs/requirements/self_review_rule.md`のチェックリスト（抜け漏れ・実現可能性）を使う
- 見つかった問題は要件定義書自体を修正してから次に進む（レビュー結果に「後で直す」を残さない）

## 3. 設計

- `docs/design/cycleN_design.md`を作成
- FRごとに、対象ファイル・変更前後のコード（diff形式推奨）を具体的に書く。「〜のように直す」
  という曖昧な記述で終わらせず、実装者（未来の自分や別セッションのAI）がそのまま手を動かせる
  粒度にする
- 設計中に既存コードを`grep`等で確認し、想定と違う箇所（未使用コード、対象外にすべき要素等）
  があれば設計書にその判断根拠ごと書く
- 実装順（どのFRから着手するか、依存関係）を明記する

### DBスキーマを新設・変更する場合は必ずセットで更新する

新しいテーブルを作る・既存テーブルにカラムを追加変更する設計をするときは、以下の3点を
**必須の作業として設計に含める**（漏れやすいので明示的にチェックする）：

1. `docs/design/table/TBL-XXX_テーブル名.md`を新規作成（新設の場合）または更新
   （変更の場合）。既存の`TBL-001`〜`TBL-003`と同じ形式（基本情報・カラム定義・
   インデックス・外部キー制約・備考）で書く
2. `docs/design/table/er_diagram.md`（Mermaid ER図）にテーブルと関係を反映する
3. `docs/design/table/table_list.md`の一覧にテーブルを追記・更新する

この3点は「設計書にDBの変更点を書いた」だけでは終わらない。テーブル単体の定義書・
ER図・一覧のいずれかが欠けたまま実装フェーズに進まないよう、設計セルフレビュー
（`req-design-traceability`）でこの3点の存在を明示的にチェック項目に含める。

### `backend/scripts/`に新しいバッチを追加する場合も必ずセットで更新する

新しいバッチ（`backend/scripts/`配下の手動・スケジュール実行スクリプト）を作る前に、
まず`docs/design/batch/batch_list.md`を確認し、既存のバッチ・共通モジュールで
対応できないか検討する（似た機能のスクリプトが「謎のスクリプトファイル」として
多重に作られることを防ぐため）。新設が必要と判断したら：

1. `docs/design/batch/BATCH-XXX_バッチ名.md`を新規作成する（画面・APIと同じ形式：
   基本情報・概要・入力出力・処理内容・依存モジュール）
2. `docs/design/batch/batch_list.md`の一覧にバッチを追記する

これも設計セルフレビュー（`req-design-traceability`）でチェック項目に含める。

## 4. 設計セルフレビュー

- `docs/design/cycleN_design_review.md`を作成
- `docs/design/self_review_rule.md`のチェックリスト（機能面・性能面・セキュリティ・拡張性）
- **続けて[req-design-traceability](../req-design-traceability/SKILL.md)を実行する**。
  FR単位の対応確認だけでは、FR内の箇条書き1つ1つの見落としや、要件文と設計判断の
  食い違いを見逃す
- 見つかった問題は要件定義書・設計書の両方を実態に合わせて修正してから次に進む

## 5. 実装

- 設計書のdiffどおりに実装する。設計にない変更（ついでの改善等）はしない
- 実装中に設計と実際のコードが食い違う・設計の想定が誤っていたと分かった場合は、
  黙って実装を優先せず、設計書を修正してから実装を続ける
- バックエンドロジックが絡む変更は、モックではなく実際のEDINETデータで動作確認する
- **外部API・システムに負荷がかかる規模の操作（数百リクエスト以上・分単位以上かかる
  バッチ実行等）は、設計書に書いてあることを理由に無許可で実行しない。実行前に
  ユーザーへ「これは外部APIに約N回アクセスします、実行してよいですか」と一言確認する**
  （小規模なテスト実行、数件〜数十件・数十秒程度で終わるものは都度確認不要）。
  同様に、データの更新・削除、アーキテクチャ・ライブラリ・方針の変更も実行前に確認する
  （実例：サイクル9でEDINETへの約3,650リクエスト規模のバックフィルを無許可で実行してしまい、
  ユーザーからこのルールを明示された）

## 6. 実装セルフレビュー

- `docs/development/cycleN_development_review.md`を作成（新規サイクルなら新規ファイル、
  既存サイクルへの追記なら該当ファイルに追記）
- `git diff`を設計書のdiffと1件ずつ突き合わせる
- 動作確認：`npx tsc -b --noEmit`・`npx oxlint`が通ること、vite HMRログにエラーが
  出ていないこと、バックエンド・フロントエンドが起動して応答すること（`curl`で確認）
- 実装中にユーザーの実機確認等で設計スコープ外の不具合が見つかった場合は、その場で
  原因調査・修正し、レビュー結果に「設計スコープ外で見つかった不具合」として追記する
- 最後に、今回スコープ外にした事項があれば`cycleX_backlog.md`に追記する
  （[todo-tracker](../todo-tracker/SKILL.md)参照）

## 7. `docs/INDEX.md`の更新

- `docs/INDEX.md`の「ディレクトリ構成」（今回新規作成したファイルを反映）と
  「現在のフェーズ」（サイクルの状態）を最新化する
- **詳細情報をINDEX.mdに書き写さない**。画面・テーブル・API等の詳細は各一覧ファイル
  （`screen_list.md`・`table_list.md`・`api_list.md`等）側で管理し、INDEX.mdからは
  リンクするだけにする（書き写すと更新漏れで陳腐化する。実際に過去そうなった）
- サイクルの区切りだけでなく、`docs/`直下に新しいディレクトリ・重要ファイルを追加した
  タイミング（例：`docs/ideas/`を新設した時）でも更新する

## 8. コミット

- **ユーザーから明示的に指示されるまでコミットしない**
- コミット前に`git status`で意図しない変更が含まれていないか確認する
- コミットメッセージは「何を」「なぜ」を簡潔に。サイクル番号・FR番号を含める
