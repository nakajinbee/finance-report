# TODO確認・積み残し管理

このプロジェクトでは、各サイクルの要件定義書に「スコープ外（次サイクル以降）」節があるが、
サイクルをまたぐと記述が埋もれて忘れられやすい。それを防ぐため、未着手事項は
**`docs/requirements/cycleX_backlog.md`に一元管理する。**

このファイルは、思いついたことを何でも書く`docs/ideas/`とは段階が違う。全体の流れは
以下の3段階：

```
docs/ideas/（検討中・やるか未確定）
   ↓ ユーザーと相談して「やる」と決まったら
docs/requirements/cycleX_backlog.md（決定済みだが未着手）
   ↓ 実際に着手するサイクルが決まったら
cycleN_requirements.md（進行中のサイクルの正式なFR）
```

各段階で、移した元からは必ず削除する（同じ項目が複数箇所に存在し続けることがないように
する）。

## 基本ルール

- `cycleX_backlog.md`は「まだどのサイクルにも割り当てられていない、やることは決まっている
  未着手事項」の唯一の一覧であり続ける。ここに載っている＝まだやっていない、という前提を崩さない
- 各項目には以下を必ず書く：
  - **内容**：何をするか
  - **出所**：どのサイクル・どの経緯で発生したか
  - （解消条件が明確なものは）**何が満たされたら着手できるか**（例：「性能問題が
    実際に報告されたら」「必要になった時点で」）
- サイクル番号が確定した具体的なタスクにする段階（＝実際に着手するサイクルの要件定義に
  組み込んだ段階）で、`cycleX_backlog.md`からは削除する。二重管理をしない

## サイクル開始時にやること

- `cycleX_backlog.md`を読み、今回のサイクルで拾える項目がないか確認する
  （[cycle-workflow](../cycle-workflow/SKILL.md)の要件定義フェーズから呼ばれる）
- 拾った項目は`cycleN_requirements.md`に正式なFRとして組み込み、`cycleX_backlog.md`から削除する
- **項目に書かれた「現状は〜」という前提を鵜呑みにせず、コードを読んで裏取りする**
  （[cycle-workflow](../cycle-workflow/SKILL.md)の「企画内容のセルフレビュー」参照）。
  `docs/ideas/`・`cycleX_backlog.md`の記述は書かれた時点のスナップショットであり、
  その後の実装で状況が変わっていても自動的には更新されない

## サイクル終了時（実装セルフレビューの最後）にやること

- 今回のサイクルで「やらないと決めたこと」「スコープ外にしたこと」を`cycleN_requirements.md`の
  「スコープ外」節に書くだけで終わらせず、**`cycleX_backlog.md`にも追記する**
- コード中・ドキュメント中に新しく「TODO：」を残した場合（例：`design_guideline.md`の
  未対応項目）も、同様に`cycleX_backlog.md`に転記する。TODOコメント単体で放置しない

## TODOを見つけたとき（サイクルの途中でも）

- コード・ドキュメント中の既存の「TODO」「後で」「未対応」といった記述を見つけたら、
  それが`cycleX_backlog.md`に載っているか確認する。載っていなければ追記する
- 逆に、`cycleX_backlog.md`の項目が実は既に解消されている（実装済みだが記録の更新漏れ）
  ことに気づいたら、その場で削除する

## 書き方の例

```markdown
### 全企業データの一括DB投入・SQLiteからの移行
- **内容**：EDINETの全上場企業データを一括でDBに投入する仕組み。あわせてSQLiteから
  他のDBへの移行を検討する
- **出所**：ユーザー要望（cycle4で発生、cycle4→cycle5で持ち越し済み、未着手）
```

「出所」に持ち越し履歴（cycle4→cycle5→…）を書いておくと、長期間放置されている項目が
一目で分かる。

## サイクルとフェーズを混同しない

「サイクル」はプロジェクト全体の要件定義〜実装の1周（トピック不問の通し番号）、
「フェーズ」は1つのアイデアを分割した、そのアイデア固有の内部工程。フェーズは
複数のアイデアがそれぞれ持ちうるため、**「フェーズ2」のように単独で呼ばず、
必ず「IDEA-01のフェーズ2」のようにアイデアIDとセットで書く**（詳細：
[docs/ideas/README.md](../../../docs/ideas/README.md)の「フェーズとサイクルの違い」）。
1つのフェーズの実装前に、追加の技術検証サイクルを別途挟むこともあり、
フェーズ番号とサイクル番号は必ずしも1対1にならない。
