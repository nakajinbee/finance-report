# 要件⇔設計トレーサビリティ突合せ

`docs/design/self_review_rule.md`の通常チェック（「要件のすべてのFRに対応する設計が
あるか」）はFR＝見出し単位の確認であり、**FRの中の箇条書き1つ1つ**まではチェックしない。
このSkillはその一段細かいレベルの突合せを行う。設計セルフレビューの最後、判定を出す前に
必ず実施する。

## やり方

1. `cycleN_requirements.md`を開き、各FRの箇条書きを1行ずつリストアップする
2. それぞれについて、`cycleN_design.md`のどこで対応しているかを具体的に確認する。
   「たぶんこの節で扱っている」ではなく、該当する具体的な記述（diff・コード例・説明文）を
   指せることを基準にする
3. 見つかった問題は次の3種類に分類する：
   - **抜け**：要件の箇条書きに対応する設計記述がどこにもない
   - **食い違い**：設計が要件と異なる判断をしている（設計判断が妥当な場合もあるが、
     その場合は要件定義書側を実態に合わせて修正する。設計だけ直して要件を放置しない）
   - **部分対応**：要件の1文の中に複数の要求が含まれ、一部だけ設計されている
     （例：「AとBを適用する」の要件に対し設計がAしか触れていない）
4. 既存コードを`grep`等で確認し、設計が「対象」とした箇所が実際に正しいかも合わせて検証する
   （フォーカス不可の要素を対象に含めていないか、対象外とすべきものを見落としていないか等）
5. 見つかった問題は、要件定義書・設計書の両方をその場で修正する。「後で直す」を残さない

## レビュー結果への書き方

`cycleN_design_review.md`に「要件トレーサビリティチェック」の節を追加し、要件の箇条書きを
1件ずつ列挙してチェック済みマークを付ける。問題が見つかった項目は「不一致を発見」として
何が食い違っていたか・どう修正したかを明記する。

```markdown
## N. 要件トレーサビリティチェック（FRの箇条書き単位で設計と突き合わせ）

- [x] FR-xx「〇〇を定義する」→ 設計§1で値まで一致
- [x] FR-xx「△△も同じ値に更新する」→ **不一致を発見**：設計は「未使用のため更新ではなく
  削除」という判断をしていたが、要件文は「更新する」のままだった。設計判断（削除）自体は
  `grep`による裏付けがあり妥当なため、要件定義書側を実態に合わせて修正した
```

## このSkillが対象にしないもの

- FR単位の大枠の対応確認（`self_review_rule.md`の通常チェックで実施済みの前提）
- 実装フェーズでの`git diff`と設計書の突き合わせ（これは実装セルフレビューの範囲。
  [cycle-workflow](../cycle-workflow/SKILL.md)参照）
