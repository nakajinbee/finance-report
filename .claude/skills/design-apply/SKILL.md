# デザイン検討・実装

このプロジェクトの本番デザインコンセプトは`docs/design/design_guideline.md`に一元化されている。
UI・見た目に関わる変更をするときは、必ずこのSkillの手順に従う。

## デザインを変更・追加するとき

1. **着手前に`docs/design/design_guideline.md`を全文読む。** 部分的に知っている記憶に
   頼らない（ガイドラインは更新され続けるため、古い記憶と食い違う可能性がある）
2. 今回やりたいことがガイドラインで既に決まっている範囲なら、その通りに実装する
3. ガイドラインに決まっていない・新しい判断が必要な項目が出てきたら、**単独で決めずに
   `AskUserQuestion`で選択肢を提示してユーザーに選んでもらう**。プレビュー付きで具体的な
   見た目の違いが伝わるようにする
   - 判断が必要になりがちな観点の例：色（ベース・アクセント・エラー等）、フォント、
     余白・角丸・シャドウ、状態変化（hover/focus/disabled）、アイコンの有無、
     ローディング・空状態の表現、バッジ・ラベル表現、グラフの装飾
4. ユーザーが選んだ内容を`design_guideline.md`に反映する（後述の「ガイドラインの書き方」参照）
5. 実装は、色・トークンの実装方式（独自CSS変数トークンか、Tailwind標準クラス直接か）を
   ガイドラインに明記されたとおりに行う。曖昧なまま実装を始めない
6. 実装後、`grep`で旧い色コード・旧トークン名・旧クラス名が残っていないか確認する
   （置き換え漏れ・二重管理の検出）

## ガイドラインの書き方（重要）

`design_guideline.md`は**常に「今の最終的な決定事項」だけを書く。時系列の記述をしない。**

- ❌「サイクル4では〇〇だったが、サイクル5で△△に変更した」
- ❌「現状は暫定実装として〜」「後で変更予定」
- ✅「〇〇は△△とする」（現在の決定をそのまま断定的に書く）

理由：このファイルは将来、別のサイクル・別のセッションのAIが「今のルールは何か」を
確認するために読む。経緯や比較が混ざっていると、どれが今も有効な決定でどれが過去の話か
判別しにくくなり、誤読・二重実装のもとになる。

- 決定の経緯（なぜこの色を選んだか、ユーザーがどの選択肢を選んだか等）は
  `docs/requirements/cycleN_requirements.md`や`docs/design/cycleN_design.md`側に書く。
  そちらは時系列の記録を残してよい（むしろ残すべき）場所
- ガイドラインに書く内容に迷ったら「これは半年後に読んでも今すぐ実装に使える情報か」を基準にする
- 未対応・将来検討の項目は「TODO：」で明記してよい（時系列の説明ではなく、今の未完了状態の記録なので）

## デザインレビューで確認すること

- 色の実装方式（独自トークンかTailwind標準クラスか）に一貫性があるか
- ガイドラインが挙げる中立色・アクセント色以外の色が、独自の判断でハードコードされていないか
- 新しく追加したUI要素（フォーム、トグル、バッジ等）の状態表現がガイドラインの規定と一致するか。
  規定がなければ実装前にガイドラインへの追記が必要
